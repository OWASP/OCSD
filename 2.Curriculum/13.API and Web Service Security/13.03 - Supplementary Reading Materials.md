# Supplementary Reading Materials - API and Web Service Security

**Audience:** Backend, full-stack, platform, DevOps, and application security engineers  
**Goal:** Help developers design, implement, test, and operate APIs that resist common real-world attacks.

APIs are the connective tissue of modern software. They expose application functionality to mobile clients, third-party integrations, partner systems, internal microservices, and public developers. Their ubiquity makes them one of the most consequential attack surfaces in contemporary application security — OWASP has maintained a dedicated API Security Top 10 since 2019, distinct from the web application Top 10, precisely because the attack patterns are different.

The key distinction: APIs deal in structured data and programmatic access rather than rendered HTML. This changes both the threat model (machine-to-machine interactions mean rate limiting and enumeration attacks are particularly significant) and the defenses (authentication tokens, schema validation, and gateway-level controls are the primary tools). Most API vulnerabilities are not exotic — they are the predictable consequence of trusting clients, failing to authorize at the resource level, or implementing authentication incorrectly.

This module covers authentication and authorization patterns for APIs, rate limiting, input validation, security considerations for REST, GraphQL, SOAP, and gRPC, webhook security, API gateway architecture, third-party API integration, and security testing. The thread connecting all sections is the same: APIs must be designed with security as a structural property, not applied as a filter at the perimeter.

## OWASP API Security Top 10 (2023)

The OWASP API Security Top 10 is a useful risk model because it maps closely to the mistakes developers make in real API implementations: exposing object IDs, trusting clients, failing to validate tokens, omitting authorization checks, allowing expensive calls, and forgetting inventory.

Authentication answers "who is this?" Authorization answers "what are they allowed to do?" The OWASP API Security Top 10 consistently lists broken object-level authorization and broken function-level authorization at the top — not because they are technically complex, but because they are consistently under-implemented.

| OWASP API Risk | Typical Interpretation | Typical Root Cause | Secure Design Habit |
|---|---|---|---|
| API1:2023 Broken Object Level Authorization (BOLA) | User can access another user's object by changing an ID. | Object lookup happens before ownership/access check. | Always authorize every object read, update, and delete. |
| API2:2023 Broken Authentication | Identity can be spoofed, bypassed, brute-forced, or replayed. | Weak token handling, missing MFA where needed, insecure session/token storage. | Centralize authn, validate tokens strictly, rate-limit login/token paths. |
| API3:2023 Broken Object Property Level Authorization | User can read or modify fields they should not access. | Mass assignment, over-broad serializers, missing field-level checks. | Use allowlists for input/output fields by role and context. |
| API4:2023 Unrestricted Resource Consumption | API can be abused to exhaust CPU, memory, storage, or paid resources. | Missing limits on request size, pagination, query complexity, file uploads, external calls. | Apply quotas, timeouts, pagination caps, and cost controls. |
| API5:2023 Broken Function Level Authorization (BFLA) | User can call functions reserved for another role. | Route exists but role/method checks are incomplete. | Protect endpoints by capability, not just UI visibility. |
| API6:2023 Unrestricted Access to Sensitive Business Flows | Bots or users automate flows such as signup, coupon use, ticket purchase, or password reset. | Business abuse not considered during threat modeling. | Add abuse detection, velocity checks, proof-of-work/friction where appropriate. |
| API7:2023 Server-Side Request Forgery (SSRF) | API fetches attacker-controlled URLs from internal networks or metadata services. | Backend URL fetchers trust user-provided destinations. | Allowlist destinations, block internal IP ranges, disable redirects, use egress controls. |
| API8:2023 Security Misconfiguration | Defaults, headers, CORS, errors, storage, or cloud settings expose risk. | Insecure defaults and inconsistent environment hardening. | Maintain secure baseline templates and config tests. |
| API9:2023 Improper Inventory Management | Unknown, old, shadow, or undocumented APIs remain exposed. | No source of truth for endpoints, owners, versions, or data classification. | Keep API inventory, owners, versions, auth schemes, and deprecation dates. |
| API10:2023 Unsafe Consumption of APIs | Your API trusts data or behavior from third-party APIs. | External responses are treated as trusted internal data. | Validate external responses, isolate failures, use timeouts and circuit breakers. |

### BOLA: Broken Object Level Authorization

BOLA is one of the most common and damaging API flaws because APIs frequently expose object identifiers in paths, query parameters, headers, and payloads.

**Insecure pattern**

```http
GET /api/v1/invoices/987654
Authorization: Bearer token-for-user-a
```

```pseudo
invoice = db.invoices.find(request.path.invoice_id)
return invoice
```

The code retrieves the object but never checks whether the authenticated user can access that invoice.

**Secure pattern**

```pseudo
user = requireAuthenticatedUser(request)
invoice = db.invoices.find(request.path.invoice_id)

if invoice is null:
    return 404

if not policy.canReadInvoice(user, invoice):
    return 404  # or 403, depending on disclosure policy

return serializeInvoiceFor(user, invoice)
```

**Developer checklist**

- Do not rely on hidden UI controls or non-guessable IDs as authorization.
- Perform authorization after loading the target object and before returning or modifying it.
- Check authorization on reads, writes, deletes, exports, downloads, and batch endpoints.
- Include tenant, organization, project, ownership, sharing, and delegated-access rules.
- Test horizontal access: user A attempts user B's object IDs.
- Test vertical access: normal user attempts admin-owned resources or admin-only actions.

Object-level authorization requires that every API endpoint verify not just that the requesting user is authenticated, but that the authenticated user is authorized to access the specific object being requested.

# Missing object-level authorization (vulnerable)
@app.get("/api/orders/{order_id}")
def get_order(order_id: int, current_user=Depends(get_current_user)):
    order = db.get_order(order_id)
    return order  # Returns any order regardless of ownership

# Correct: object-level authorization enforced
@app.get("/api/orders/{order_id}")
def get_order(order_id: int, current_user=Depends(get_current_user)):
    order = db.get_order(order_id)
    if order is None or order.user_id != current_user.id:
        raise HTTPException(status_code=404)  # 404 prevents ID enumeration
    return order

Return 404, not 403, for objects the user should not know exist. 403 confirms the object exists but is inaccessible — enabling enumeration of valid IDs. 404 provides no confirmation, preventing enumeration.

Implement authorization checks consistently. Use a centralized authorization function applied uniformly rather than per-handler checks that may be accidentally omitted. Authorization-by-default (deny unless explicitly granted) is safer than authorization-on-request.

Never rely on client-provided IDs for scoping. An API that accepts user_id as a request parameter and uses it to scope queries is trivially bypassed by changing the value. Always derive the acting user's identity from the validated authentication token.

### Broken Authentication

Authentication establishes who the caller is. Authorization decides what that caller can do. Mixing the two often leads to bugs.

Common authentication failures:

- Accepting unsigned or weakly signed tokens.
- Not validating token issuer, audience, expiry, or algorithm.
- Storing API keys or tokens in logs, URLs, mobile apps, or browser storage without controls.
- Missing rate limits on login, token refresh, OTP, password reset, or API-key validation paths.
- Long-lived tokens with no rotation, revocation, or anomaly detection.

### Broken Object Property Level Authorization

This includes both **excessive data exposure** and **mass assignment**.

**Insecure mass assignment**

```json
PATCH /api/users/me
{
  "displayName": "Shruti",
  "role": "admin",
  "billingPlan": "enterprise"
}
```

```pseudo
user.update(request.body)  // dangerous: accepts fields the user should not control
```

**Secure approach**

```pseudo
allowed = pick(request.body, ["displayName", "timezone", "avatarUrl"])
user.update(allowed)
```

For output, avoid returning entire ORM models. Create response DTOs or serializers that are specific to the caller's role and context.

### Unrestricted Resource Consumption

Attackers do not always need to bypass authorization. Sometimes they can simply make valid requests that are too expensive.

Controls to implement:

- Request body size limits.
- File upload size, type, and scanning limits.
- Pagination maximums.
- Query timeout limits.
- GraphQL depth and complexity limits.
- Per-user, per-token, per-IP, and per-tenant quotas.
- External API timeout and retry budgets.
- Async job queue limits.
- Cost controls for SMS, email, AI, search, payment, and cloud resources.

### BFLA: Broken Function Level Authorization

BFLA happens when endpoints or HTTP methods are callable by roles that should not have that function.

**Examples**

- `POST /admin/users/{id}/disable` callable by normal users.
- `DELETE /api/projects/{id}` allowed to project viewers.
- `PUT /api/accounts/{id}/plan` allowed to account members instead of billing admins.
- `GET /internal/reports` exposed externally.

**Secure pattern**

Define capabilities such as `project.delete`, `user.disable`, `invoice.refund`, and `gateway.configure`. Map roles to capabilities, then enforce capabilities at endpoint boundaries and service-layer methods.

Function-level authorization controls which users can invoke which endpoints or operations.

Role-based endpoint access restricts endpoints based on the user's role. Administrative endpoints must reject non-administrative accounts regardless of authentication.

def require_role(required_role: str):
    def dependency(current_user=Depends(get_current_user)):
        if required_role not in current_user.roles:
            raise HTTPException(status_code=403)
        return current_user
    return dependency

@app.delete("/admin/users/{user_id}")
def delete_user(user_id: int, admin=Depends(require_role("admin"))):
    ...

HTTP method restrictions per role. A read-only user should be able to GET but not PUT, PATCH, or DELETE. Enforce method-level restrictions as part of authorization.

Administrative function protection: Rate limit administrative endpoints more aggressively; require re-authentication or MFA for destructive operations; log all administrative actions with full context.

### Unrestricted Access to Sensitive Business Flows

These issues are often invisible to scanners because each request is technically valid.

Examples:

- Automated account creation.
- Credential stuffing.
- Coupon abuse.
- Inventory hoarding.
- Password reset spam.
- Scraping public but commercially sensitive listings.
- Repeated free-trial creation.

Mitigations:

- Velocity limits by account, device, IP, ASN, payment method, and tenant.
- Bot detection and step-up challenges.
- Idempotency and duplicate detection.
- Fraud rules for high-risk flows.
- Business event logging.
- Human review for suspicious patterns.

### SSRF in APIs

SSRF occurs when an API causes the server to make requests to attacker-controlled destinations.

High-risk features:

- Webhook registration URLs.
- Image import by URL.
- PDF generation from URL.
- URL preview or metadata fetch.
- Data import connectors.
- Server-side proxy endpoints.

Defensive controls:

- Use destination allowlists where possible.
- Resolve DNS and validate final IP addresses before connecting.
- Block private, loopback, link-local, multicast, and cloud metadata IP ranges.
- Re-check destination after redirects; ideally disable redirects.
- Restrict outbound network egress from application workloads.
- Enforce low timeouts and response size limits.
- Never return raw internal error details to clients.

### Security Misconfiguration

API misconfiguration often happens outside application code.

Examples:

- Debug mode enabled.
- Verbose stack traces returned.
- Wildcard CORS with credentials.
- Missing HTTPS redirect or HSTS.
- Public cloud storage buckets.
- Default admin credentials.
- Overly broad gateway routes.
- Missing `Content-Type` enforcement.

### Improper Inventory Management

You cannot protect what you do not know exists.

Inventory should include:

- Endpoint path, method, version, owner, repository, and runtime service.
- Auth scheme and required scopes/roles.
- Data classification.
- Public/internal/admin status.
- Deprecation date and migration plan.
- Upstream and downstream dependencies.
- Related OpenAPI/GraphQL/gRPC schema.

### Unsafe Consumption of APIs

Treat external API responses as untrusted input.

Controls:

- Validate response schema.
- Apply allowlists for enum values and URLs.
- Escape or encode data before rendering.
- Use timeouts, retries with backoff, and circuit breakers.
- Avoid forwarding third-party error messages directly to users.
- Store third-party tokens securely and scope them minimally.
- Monitor dependency failures and data anomalies.

## 13.2 API Authentication

Authentication establishes who is making an API request. For APIs, this is more complex than for browser-based applications: API clients may be end users, automated services, mobile applications, or other servers. Each scenario warrants a different authentication mechanism, and the choice of mechanism has significant security implications.

### OAuth 2.0

OAuth 2.0 is an authorization framework used to grant limited access to HTTP services. It is not, by itself, an authentication protocol; OpenID Connect adds an identity layer on top of OAuth 2.0.

OAuth 2.0 is the standard framework for delegated authorization — allowing an application to access resources on behalf of a user without the user sharing their credentials with the application. It is the foundation of most modern API authentication for user-facing applications.

#### Authorization Code Flow with PKCE

Use Authorization Code with PKCE for browser-based apps, mobile apps, desktop apps, and increasingly for confidential web apps as well.

Simplified flow:

1. Client generates a high-entropy `code_verifier`.
2. Client derives a `code_challenge`, preferably using `S256`.
3. Client sends the user to the authorization server with the `code_challenge`.
4. User authenticates and consents.
5. Authorization server returns an authorization code.
6. Client exchanges the code plus `code_verifier` for tokens.
7. Authorization server verifies that the verifier matches the original challenge.

Security notes:

- Prefer `S256` over `plain`.
- Use exact redirect URI matching.
- Validate `state` to prevent CSRF.
- Use `nonce` when OpenID Connect ID tokens are involved.
- Do not expose tokens in URLs.
- Do not use implicit flow for modern applications.

The Authorization Code flow with Proof Key for Code Exchange (PKCE) is the recommended OAuth 2.0 flow for applications that interact with users — web applications, mobile applications, and Single Page Applications (SPAs).

The flow in a different way:

The client generates a random code_verifier (a cryptographically random string of 43–128 characters) and derives a code_challenge: code_challenge = BASE64URL(SHA256(code_verifier)).
The client redirects the user to the authorization server with the code_challenge and code_challenge_method=S256.
The user authenticates and consents at the authorization server.
The authorization server redirects back to the client with an authorization code.
The client exchanges the authorization code for tokens by presenting both the code and the original code_verifier. The authorization server verifies that SHA256(code_verifier) matches the stored code_challenge.


Why PKCE? The original Authorization Code flow was vulnerable to authorization code interception: if an attacker could intercept the authorization code (through a malicious redirect URI, a shared scheme on mobile, or a browser redirect leak), they could exchange it for tokens. PKCE binds the code exchange to the original initiating party — only the client that generated the code_verifier can successfully exchange the code.

PKCE is required for public clients (mobile apps, SPAs) that cannot securely store a client secret. It is also recommended for confidential clients (server-side web applications) as defense-in-depth.

Security requirements for Authorization Code flow:

Register exact redirect URIs; never use partial matching or wildcards.
Validate the state parameter on the redirect callback to prevent CSRF.
Use short-lived authorization codes (typically 1–10 minutes).
Authorization codes must be single-use — the authorization server must reject reuse.

#### Client Credentials Flow

Use client credentials for service-to-service access where no end user is present.

Implementation guidance:

- Authenticate the client using a strong method such as private key JWT, mTLS, or securely stored client secrets.
- Issue short-lived access tokens.
- Scope tokens to the minimum required service actions.
- Rotate credentials.
- Monitor unusual token use by client, environment, region, and endpoint.

The Client Credentials flow is for machine-to-machine authentication where there is no user involved. A backend service authenticating to another backend service uses its own credentials (client ID and client secret) to obtain an access token directly from the authorization server.

POST /token
grant_type=client_credentials&
client_id=service-a&
client_secret=<secret>&
scope=read:data write:records

The client secret must be stored securely (in a secrets manager, not in environment files checked into version control) and rotated regularly. Each service should have its own client credentials — shared credentials between services make rotation difficult and prevent per-service audit trails.

Token Validation and Verification

Every API endpoint that accepts OAuth tokens must validate them on every request:


Signature verification: Verify the token signature against the authorization server's public key, fetched from the JWKS (JSON Web Key Set) endpoint and cached with a reasonable TTL.
Expiry check: Reject tokens where exp (expiry time) is in the past, with a small clock skew tolerance of no more than a few seconds.
Issuer check: Verify the iss (issuer) claim matches the expected authorization server.
Audience check: Verify the aud (audience) claim includes this API's identifier. This prevents token confusion attacks where a token issued for one service is used against another.
Scope check: Verify the token has the scopes required for the specific operation being performed.


Token introspection (RFC 7662) provides an alternative: the API calls the authorization server's introspection endpoint and receives a response indicating whether the token is active and its claims. This adds latency but provides real-time revocation status.

Scope Management and Enforcement

Scopes limit what an access token permits. Scopes must be:


Granular enough to enforce meaningful least privilege. A single api:access scope for an entire API is effectively no scope limitation.
Enforced at the endpoint level, not just validated at the perimeter. The API gateway may verify the token is valid; each endpoint must verify the token has the required scope for its specific operation.
Displayed to users during the consent screen so they understand what access they are granting.

#### Token Validation and Verification

Every API receiving bearer tokens should validate:

- Signature.
- Algorithm.
- Key ID and trusted key source.
- Expiration (`exp`).
- Not-before (`nbf`), when used.
- Issuer (`iss`).
- Audience (`aud`).
- Required scopes or permissions.
- Tenant or organization context, if applicable.

Never accept a token merely because it decodes successfully.

#### Scope Management and Enforcement

Scopes should represent capabilities, not vague user roles.

Poor scopes:

```text
user
admin
api
full_access
```

Better scopes:

```text
invoice:read
invoice:write
project:delete
webhook:manage
billing:refund
```

Scopes are not a substitute for object-level authorization. A token with `invoice:read` still should not read every invoice.

### API Keys

API keys are identifiers and secrets used to authenticate calling applications or users. They are simple, but easy to misuse.

API keys are long-lived credentials used for authenticating API clients, particularly server-to-server integrations and developer access.

#### Secure Key Generation

- Use a cryptographically secure random generator.
- Use enough entropy to resist guessing.
- Prefix keys with non-secret metadata for identification, such as `sk_live_`, `sk_test_`, or key ID prefixes.
- Store only a hash of the secret portion when possible.
- Show full key only once at creation time.

#### Secure Key Storage

- Store server-side keys in a secrets manager.
- Never hard-code production keys in source code.
- Never ship privileged keys in browser or mobile clients.
- Avoid logging keys in request headers, query strings, or error reports.
- Use secret scanning in repositories and CI/CD.

Generate API keys using a cryptographically secure random number generator with at minimum 128 bits, preferably 256 bits of entropy, encoded as a hex or base62 string.

Store only a hash of the API key in the database, not the plaintext key. The key is displayed to the user once at generation time; the database stores SHA-256(key) for comparison on subsequent requests. If the database is compromised, stored key hashes do not provide the attacker with valid credentials.

python:
import secrets
import hashlib

def generate_api_key() -> tuple[str, str]:
    """Returns (plaintext_key, key_hash) — store only the hash."""
    key = secrets.token_hex(32)  # 64 character hex = 256 bits
    key_hash = hashlib.sha256(key.encode()).hexdigest()
    return key, key_hash

def verify_api_key(provided_key: str, stored_hash: str) -> bool:
    computed_hash = hashlib.sha256(provided_key.encode()).hexdigest()
    return secrets.compare_digest(computed_hash, stored_hash)

#### Key Rotation Policies

A mature key lifecycle includes:

- Creation with owner and purpose.
- Scope assignment.
- Last-used tracking.
- Rotation windows.
- Emergency revocation.
- Expiry for temporary keys.
- Alerts for old, unused, or over-privileged keys.

Define a maximum lifetime for API keys and enforce rotation. Key rotation should support a graceful transition period where both the old and new key are valid, allowing the consumer to update their integration without an outage. Log every key rotation event, including when the old key was last used.

Key Scope and Permission Limiting

API keys should carry permission metadata restricting what the key can do. A key created for read-only access should not be able to write or delete. Support key restriction by IP address (allowlisting originating IP ranges) and by API path prefix where the use case permits.

#### Avoiding Exposure in URLs and Logs

Do not use API keys in query parameters:

```http
GET /api/data?api_key=secret
```

Query strings often appear in logs, analytics, browser history, proxies, and referrers.

Prefer headers:

```http
Authorization: Bearer <token>
X-API-Key: <key>
```

Even with headers, configure log redaction.

API keys must never appear in URLs. Pass API keys in HTTP headers (Authorization: Bearer <key> or X-API-Key: <key>). URLs are logged in web server logs, proxy logs, browser history, and referrer headers — all less protected than the application itself.

Scrub API keys from all logs at the logging layer. The logging framework should be configured to automatically mask fields named api_key, x-api-key, or authorization.

### JWT: JSON Web Tokens

A JWT is a compact token format. Security depends on how it is signed, validated, issued, stored, and revoked.

JWTs are a compact, self-contained token format widely used for API authentication. A JWT carries claims in a signed payload, allowing the API to validate the token without calling back to the authorization server.

#### Token Structure

A signed JWT has three base64url-encoded parts:

```text
header.payload.signature
```

- Header: token type and algorithm.
- Payload: claims.
- Signature: integrity protection.

#### Algorithm Selection: RS256 vs HS256

- `HS256` uses a shared secret. Any service that can verify can also sign if it has the secret.
- `RS256` uses asymmetric keys. APIs can verify with a public key while the issuer keeps the private key.

For distributed systems, asymmetric signing such as `RS256` or `ES256` is often easier to operate safely because verifiers do not need signing secrets.

JWT validation must explicitly allow expected algorithms. Never trust the `alg` header alone.

A JWT consists of three Base64URL-encoded parts: header.payload.signature. The header specifies the algorithm and token type. The payload contains claims. The signature covers the header and payload.

Validation sequence:


Parse the three parts and verify the alg header matches an algorithm your application explicitly permits.
Verify the signature using the correct key.
Verify exp is in the future.
Verify iat (issued at) is in the past.
Verify iss matches the expected issuer.
Verify aud includes the expected audience.
Verify any additional application-specific claims.


Use a well-tested JWT library — do not implement JWT parsing or signature verification from scratch. JWT validation has many subtle failure modes.

Algorithm Selection: RS256 vs HS256

HS256 (HMAC with SHA-256) uses a shared symmetric secret. Both the issuer and every verifier must possess the same secret. Appropriate for a single application that both issues and verifies its own tokens. Not appropriate for tokens issued by a central authorization server verified by multiple independent services — sharing the symmetric secret with every service means any compromised service can forge tokens.

RS256 (RSA signature with SHA-256) uses an asymmetric key pair. The authorization server signs with its private key. Any service verifies with the corresponding public key. The private key never leaves the authorization server; compromise of a verification service does not enable token forgery.

ES256 (ECDSA with P-256 and SHA-256) provides similar security to RS256 with smaller key sizes and signatures. Preferred for new implementations where the library supports it.

Critical — algorithm confusion attacks: Never accept the algorithm from the token header without validation. The alg: none attack strips the signature and allows payload modification. The HS256/RS256 confusion attack uses the public RSA key as an HMAC secret, which an attacker can obtain from the JWKS endpoint. Always specify the allowed algorithm explicitly:

python:
import jwt

payload = jwt.decode(
    token,
    public_key,
    algorithms=["RS256"],  # Explicitly allowed — rejects 'none', 'HS256', etc.
    audience="https://api.example.com",
    issuer="https://auth.example.com"
)

#### Claims Validation

Validate at least:

- `exp`: token has not expired.
- `iat`: token was issued at a reasonable time.
- `nbf`: token is valid now, if present.
- `iss`: token came from the expected issuer.
- `aud`: token was issued for this API.
- `sub`: subject is present and valid.
- `scope` or permissions: caller has required capability.
- tenant/org claims: match the route or resource context.

Claims Validation


exp (Expiration Time): Reject tokens past this time. Short-lived tokens (15–60 minutes) limit the damage window if compromised.
iat (Issued At): Verify the token was issued in the past. Future iat values indicate clock skew or manipulation.
iss (Issuer): Verify the expected authorization server issued the token. Prevents accepting tokens from an attacker-controlled issuer.
aud (Audience): Verify the token was intended for this specific API. Prevents token replay across services.
jti (JWT ID): A unique identifier for the token. Track used jti values to prevent replay attacks.

#### Refresh and Revocation

Access tokens should usually be short-lived. Refresh tokens require stronger controls:

- Store refresh tokens securely.
- Rotate refresh tokens on use.
- Detect refresh-token reuse.
- Revoke refresh tokens on password change, compromise, logout, or risk events.
- Maintain a revocation or session state mechanism when immediate invalidation is required.

Refresh token rotation: Issue a new refresh token every time one is used. The old token is immediately invalidated. Double-use of a refresh token (by a legitimate client and a thief) is detectable, allowing the authorization server to revoke the entire token family.

Revocation options: Short expiry times (the token becomes useless quickly), a token revocation blocklist checked on each request, or token introspection (RFC 7662) for real-time status.

#### Avoid Sensitive Data in JWT Payloads

JWT payloads are usually encoded, not encrypted. Anyone who obtains the token can read the claims unless encryption is used.

Avoid putting these in JWTs:

- Passwords or password hashes.
- API secrets.
- Personal data not needed by the API.
- Payment or health information.
- Authorization decisions that can become stale for long periods.

JWTs are signed but not encrypted by default. The payload is Base64URL-encoded and readable by anyone who possesses the token. Do not include passwords, PII, payment data, or health information in JWT claims. Use JWE (JSON Web Encryption) if sensitive claims are unavoidable. Use an opaque identifier (UUID) for the sub claim rather than a meaningful value like an email address.

## 13.3 API Authorization

Authentication answers "who is this?" Authorization answers "what are they allowed to do?" The OWASP API Security Top 10 consistently lists broken object-level authorization and broken function-level authorization at the top — not because they are technically complex, but because they are consistently under-implemented.

### Object-Level Authorization

Object-level authorization answers: **Can this caller access this specific resource right now?**

Implementation pattern:

```pseudo
function getProject(user, projectId):
    project = projectRepository.find(projectId)
    if project is null:
        return NotFound

    authorize(user, "project.read", project)
    return project
```

Best practices:

- Centralize policy logic.
- Enforce in service layer, not only controllers.
- Include tenant boundaries.
- Validate parent-child relationships. For example, `/orgs/{orgId}/projects/{projectId}` should verify that the project belongs to the org.
- Apply checks to batch operations and search results.
- Filter result sets by authorization, not just by query parameters.

### Avoid Relying on Client-Provided IDs

Client-provided IDs are useful for routing, but untrusted for access control.

**Dangerous assumption**

```pseudo
// Request says accountId=123, so the user must belong to account 123.
accountId = request.body.accountId
```

**Safer approach**

```pseudo
account = accountRepository.find(request.body.accountId)
authorize(user, "account.use", account)
```

### Function-Level Authorization

Function-level authorization answers: **Can this caller perform this operation?**

Use it for:

- Admin endpoints.
- Billing and refund operations.
- User management.
- Data export.
- API-key creation.
- Webhook management.
- Feature-flag or tenant configuration.

Also enforce HTTP method restrictions. A user who can `GET /projects/{id}` does not automatically get `DELETE /projects/{id}`.

### Attribute-Based Access Control (ABAC)

ABAC evaluates access using attributes about the user, resource, action, and environment.

Examples:

```text
Allow invoice.refund when:
- user.role == "billing_admin"
- invoice.status == "paid"
- invoice.amount <= user.refundLimit
- user.tenantId == invoice.tenantId
- request.riskScore < threshold
```

Useful attributes:

- User role, department, tenant, employment status.
- Resource owner, tenant, classification, status.
- Action type, HTTP method, sensitivity.
- Time, IP, device posture, geography, risk score.

ABAC grants access based on attributes of the subject, resource, action, and environment — enabling authorization policies that cannot be expressed as simple role checks.

Example: "A user with the analyst role can read documents that are not-confidential and belong to their assigned_projects, only during business_hours."

Policy enforcement points (PEPs) in API gateways. Open Policy Agent (OPA) at the gateway layer allows authorization policy to be written in Rego, versioned, tested, and deployed independently of application code. The application handler receives only requests that have already been authorized.


### Policy Enforcement Points in API Gateways

API gateways can enforce coarse-grained access policies, such as valid token, scope, route, and rate limit. Application services still need fine-grained object and business authorization because the gateway usually lacks full resource context.

Good division of responsibility:

| Layer | Best For | Not Enough For |
|---|---|---|
| API Gateway | Token verification, route scopes, quotas, WAF, mTLS. | Object ownership and deep business rules. |
| Service Layer | Object-level authorization, field-level authorization, business logic. | Global traffic controls alone. |
| Data Layer | Tenant filters, row-level security, safe queries. | Complete user intent and workflow context. |

## 13.4 Rate Limiting and Throttling

Rate limiting protects availability, fairness, cost, and business flows.
Rate limiting controls request frequency. It prevents abuse of API functionality, limits brute force and credential stuffing attacks, prevents denial of service from single clients, and enforces fair usage.

### Implementation Strategies

#### Per-User Limits

Useful for authenticated APIs.

Examples:

```text
100 requests/minute per user
10 password reset attempts/hour per account
5 API key creations/day per user
```

Per-user rate limits are applied to the authenticated user identity. They prevent any single user from dominating capacity and are effective against authenticated brute force. They must be applied after authentication.

#### Per-Endpoint Limits

Different endpoints have different costs and risk.

Examples:

```text
GET /products: 1000/minute
POST /login: 10/minute
POST /exports: 3/hour
POST /ai/generate: quota-based by token cost
```
Per-endpoint rate limits reflect the different cost profiles and abuse risks of different endpoints. Authentication endpoints (login, password reset, token refresh) warrant tighter limits than read endpoints.

#### Global Limits

Global limits protect the whole platform during spikes or attacks.

Examples:

- Maximum requests per region.
- Maximum concurrent requests per service.
- Maximum queued jobs.
- Maximum outbound calls to a third-party provider.

Global rate limits cap total API throughput regardless of identity, protecting against distributed attacks where load is spread across many clients.

### Fixed Window vs Sliding Window

| Algorithm | How It Works | Strengths | Weaknesses |
|---|---|---|---|
| Fixed window | Count requests in a fixed time bucket. | Simple and fast. | Boundary bursts can double effective rate. |
| Sliding log | Track each request timestamp. | Accurate. | Higher memory cost. |
| Sliding window counter | Weighted count across current and previous window. | Good balance. | Approximate. |
| Token bucket | Tokens refill over time; requests consume tokens. | Allows controlled bursts. | Requires careful tuning. |
| Leaky bucket | Requests drain at a fixed rate. | Smooths traffic. | Can add latency or reject bursts. |

Fixed window counts requests within fixed time intervals and has a boundary vulnerability (bursting across window boundaries to send double the limit in a short period). Sliding window maintains a rolling count, eliminating boundary exploitation at the cost of slightly higher implementation complexity. Token bucket allows controlled bursting: tokens accumulate at a fill rate up to a maximum and are consumed per request.

### Quota Management

Quotas often represent business tiers.

Example quota model:

```text
Free: 1,000 requests/day, burst 20/minute
Pro: 100,000 requests/day, burst 300/minute
Enterprise: custom daily quota, burst 2,000/minute
```

Design considerations:

- Define whether quotas reset by UTC day, rolling 24 hours, or billing period.
- Track usage by key, user, tenant, application, or plan.
- Provide usage dashboards.
- Alert before quota exhaustion.
- Apply different quotas to expensive operations.

API usage quotas and tiers define long-term usage allowances distinct from per-minute rate limits. A free tier might allow 1,000 requests per day; a paid tier might allow 100,000.

Rate limit headers communicate limit status to clients:

X-RateLimit-Limit: 1000
X-RateLimit-Remaining: 247
X-RateLimit-Reset: 1699200000
Retry-After: 30

### Rate Limit Headers

Legacy APIs commonly expose:

```http
X-RateLimit-Limit: 1000
X-RateLimit-Remaining: 42
X-RateLimit-Reset: 1710000000
```

Newer standardization work uses non-`X-` `RateLimit-*` style headers. Regardless of exact header set, document semantics clearly.

Also use:

```http
HTTP/1.1 429 Too Many Requests
Retry-After: 60
```
Return HTTP 429 (Too Many Requests) when a limit is exceeded. Include Retry-After so the client knows when to retry.


### Protection Against Abuse

#### Denial of Service Prevention

- Limit request size.
- Limit response size.
- Limit concurrency.
- Apply timeouts.
- Cap expensive filters, sorts, and joins.
- Cache safe repeated reads.
- Defer heavy work to async queues with quotas.

#### Credential Stuffing Mitigation

- Rate-limit login attempts by account, IP, device, ASN, and credential pair.
- Detect password spraying across accounts.
- Use MFA or step-up authentication for risky attempts.
- Avoid revealing whether username or password was wrong.
- Monitor leaked credential patterns.

Credential stuffing mitigation: Per-IP and per-user rate limits on authentication endpoints are the primary technical mitigation. Supplement with CAPTCHA challenges triggered by rate limit thresholds, device fingerprinting, and anomaly detection on login failure patterns.

#### Brute Force Protection

- Lock or slow down high-risk attempts without creating easy account lockout denial-of-service.
- Use exponential backoff.
- Protect OTP, password reset, invitation, and API-key verification endpoints.

Brute force protection: Prefer progressive delays (increasing wait time between attempts) over hard account lockout, which can be weaponized to deny service to legitimate users. CAPTCHA challenges are a less disruptive alternative to lockout.

## 13.5 Input Validation for APIs

Input validation ensures the API receives data in expected types, formats, ranges, and structures. It does not replace authorization or output encoding.

### Request Validation

#### Schema Validation

Use JSON Schema, OpenAPI, Protobuf schemas, or framework validators to enforce:

- Required fields.
- Types.
- String lengths.
- Numeric ranges.
- Enum values.
- Object shape.
- Array lengths.
- Pattern formats.
- Additional property restrictions.

Example JSON Schema fragment:

```json
{
  "type": "object",
  "required": ["email", "displayName"],
  "additionalProperties": false,
  "properties": {
    "email": { "type": "string", "format": "email", "maxLength": 254 },
    "displayName": { "type": "string", "minLength": 1, "maxLength": 80 }
  }
}
```
Schema validation using OpenAPI/JSON Schema rejects requests that do not conform to the documented structure before they reach application handlers:

yaml
paths:
  /users:
    post:
      requestBody:
        content:
          application/json:
            schema:
              type: object
              required: [username, email]
              properties:
                username:
                  type: string
                  minLength: 3
                  maxLength: 50
                  pattern: '^[a-zA-Z0-9_]+$'
                email:
                  type: string
                  format: email
                  maxLength: 254
              additionalProperties: false

additionalProperties: false is critical — it prevents clients from submitting undeclared fields that could bypass application logic.

#### Content-Type Validation

Reject unexpected content types.

```http
Content-Type: application/json
```

Do not parse JSON from `text/plain` or XML from arbitrary content types unless explicitly required and safely configured.

Content-Type validation: Verify the Content-Type header matches the expected body format. Processing a JSON body with a form parser produces unexpected behavior and potential security issues.

#### Size Limits

Set limits for:

- Total request body.
- Individual fields.
- Array length.
- Nested object depth.
- File upload size.
- Multipart part count.

Size limits: Enforce maximum request body size at the web server or gateway layer (rejecting before reaching the application) and validate individual field lengths per schema.

#### Parameter Type and Format Validation

Validate:

- UUIDs.
- Dates and timestamps.
- Currency values.
- Email addresses.
- URLs.
- Slugs.
- Page sizes.
- Sort fields.
- Filter operators.

For sorting and filtering, use allowlists:

```pseudo
allowedSortFields = ["createdAt", "updatedAt", "name"]
if request.query.sort not in allowedSortFields:
    return 400
```
Parameter type and format validation: Path parameters, query parameters, and headers must be validated for type and format before use. A UUID path parameter should be validated as a UUID, not passed as a raw string.


### Data Sanitization

Sanitization changes data. Validation accepts or rejects data. Prefer validation and safe APIs before attempting sanitization.

#### SQL Injection Prevention

Use parameterized queries or ORM query binding.

**Unsafe**

```pseudo
sql = "SELECT * FROM users WHERE email = '" + request.email + "'"
```

**Safe**

```pseudo
db.query("SELECT * FROM users WHERE email = ?", [request.email])
```
SQL injection prevention in API queries is identical to web application SQL injection prevention: parameterized queries exclusively. The API layer does not change this requirement.

#### NoSQL Injection Prevention

Be careful when request bodies are passed directly into query operators.

**Unsafe**

```json
{ "email": { "$ne": null }, "password": { "$ne": null } }
```

Mitigations:

- Enforce expected scalar types.
- Reject keys beginning with `$` where not explicitly allowed.
- Use query builders safely.
- Avoid passing raw request objects to database queries.

NoSQL injection prevention: MongoDB and other NoSQL databases are vulnerable to operator injection if user input can supply query operators:

javascript
// Vulnerable: attacker supplies username: { $gt: "" }
const user = await User.findOne({ username, password });
// Query: { username: { $gt: "" }, password: "anything" } — matches any user

// Safe: validate that username is a string, not an object
if (typeof username !== 'string') throw new ValidationError();
const user = await User.findOne({ username: String(username), password });

#### Command Injection Prevention

Avoid shell execution. If unavoidable:

- Use argument arrays, not shell strings.
- Allowlist commands and arguments.
- Run with least privilege.
- Set timeouts and resource limits.

Command injection prevention: Never construct shell commands from API request data. Use argument-array APIs, not shell strings; validate all inputs that influence system calls against strict allowlists.

### Output Encoding

APIs often return data consumed by browsers, mobile apps, logs, dashboards, or admin consoles. Output must be safe for its context.

Controls:

- Set accurate response `Content-Type`.
- Use `application/json` for JSON.
- Add `X-Content-Type-Options: nosniff` for browser-facing APIs.
- Escape data when embedding API output into HTML.
- Do not return secrets, stack traces, SQL errors, or internal service names in error responses.

Response Content-Type headers: Every API response must include an accurate Content-Type header. Omitting it from error responses can allow browsers to interpret JSON error content as HTML.

Preventing XSS in API responses: JSON encoding automatically escapes HTML-special characters in most JSON libraries. For APIs whose responses are embedded in HTML templates, apply HTML encoding explicitly.

Avoiding sensitive data in error messages: Return structured error responses with generic messages and error codes. Log detailed diagnostic information internally; never return stack traces, database error messages, query text, or internal hostnames in API responses.

## 13.6 REST API Security

### Secure Design Principles

#### Use Proper HTTP Methods

| Method | Intended Use | Security Notes |
|---|---|---|
| GET | Read resource. | Should not change server state; avoid sensitive data in query strings. |
| POST | Create or trigger operation. | Protect against replay and duplicate submission when needed. |
| PUT | Replace resource. | Validate full object and authorization. |
| PATCH | Partial update. | Use field allowlists; prevent mass assignment. |
| DELETE | Delete resource. | Require strong authorization and audit logging. |

Use proper HTTP methods semantically. GET must not have side effects — a GET /users/delete?id=123 endpoint can be triggered by an <img src=""> tag, bypassing CSRF protections that target POST. POST creates, PUT replaces, PATCH updates, DELETE removes.

#### Stateless Authentication

REST APIs often use bearer tokens. Stateless does not mean unmanaged. You still need:

- Token validation.
- Short expirations.
- Revocation strategy for high-risk contexts.
- Refresh token controls.
- Audit logs.

Stateless authentication. Authenticate each request independently using the token presented in that request. Server-side session state creates coupling between requests and complicates horizontal scaling.

#### HTTPS Enforcement

- Redirect HTTP to HTTPS where safe.
- Use HSTS for browser-facing domains.
- Avoid mixed-content API calls.
- Use modern TLS configuration.
- Consider mTLS for internal or partner APIs.

HTTPS enforcement. All API traffic requires TLS. Redirect or reject HTTP connections. Never disable certificate validation in API clients, even in development.

#### Proper Status Code Usage

| Status | Use |
|---|---|
| 200 | Successful read or operation. |
| 201 | Resource created. |
| 202 | Accepted for async processing. |
| 204 | Successful operation with no body. |
| 400 | Invalid request syntax or validation failure. |
| 401 | Missing or invalid authentication. |
| 403 | Authenticated but not authorized. |
| 404 | Resource not found; sometimes used to avoid disclosing existence. |
| 409 | Conflict, such as duplicate or state mismatch. |
| 422 | Semantically invalid input, where used by the framework/API style. |
| 429 | Rate limit exceeded. |
| 500 | Unexpected server error. |

Proper HTTP status codes: 200 (success), 201 (created), 400 (bad request), 401 (not authenticated), 403 (not authorized), 404 (not found), 422 (validation failed), 429 (rate limited), 500 (server error). Incorrect status codes interfere with monitoring and client error handling.

### Security Headers

For browser-accessible APIs:

```http
Content-Type: application/json
X-Content-Type-Options: nosniff
Cache-Control: no-store
```

Use `Cache-Control: no-store` for sensitive responses such as account data, tokens, reports, and exports.

X-Content-Type-Options: nosniff: Include on all API responses to prevent MIME type sniffing.

Cache-Control: no-store, no-cache, must-revalidate, private

### CORS Configuration

CORS is a browser security control. It is not an API authorization mechanism.

Secure CORS guidance:

- Allow only trusted origins.
- Do not use wildcard origins with credentials.
- Validate origins exactly, not with naive substring checks.
- Limit allowed methods and headers.
- Cache preflight results for reasonable periods.
- Avoid reflecting arbitrary `Origin` values.

**Dangerous**

```http
Access-Control-Allow-Origin: *
Access-Control-Allow-Credentials: true
```

### Endpoint Security

#### Versioning Strategies

Options:

```text
/api/v1/users
/api/v2/users
Accept: application/vnd.example.v2+json
```

Security considerations:

- Track old versions in inventory.
- Apply the same security controls to all supported versions.
- Set deprecation timelines.
- Remove or isolate legacy endpoints.

Versioning: API versioning allows introducing breaking security changes without immediately breaking existing clients. Remove deprecated versions on a defined schedule — they represent technical debt that may contain unaddressed vulnerabilities.

#### Resource Naming Conventions

Use predictable, consistent resource names, but never rely on obscurity for security.

#### Pagination and Filtering Security

- Enforce maximum `limit`.
- Use stable pagination, such as cursor pagination, for large datasets.
- Apply authorization filters before pagination.
- Allowlist filter fields and operators.
- Avoid exposing internal field names unnecessarily.

Pagination and filtering security: Enforce pagination limits to prevent unbounded result set requests. Apply authorization filters before pagination — never paginate then filter. Validate filtering parameters before incorporating them into database queries.

## 13.7 GraphQL Security

GraphQL gives clients flexible querying power. That flexibility must be constrained.
GraphQL exposes a single endpoint accepting arbitrarily complex queries. This flexibility creates security challenges absent in traditional REST APIs.

### Query Security

#### Query Depth Limiting

Prevent deeply nested queries:

```graphql
query {
  user {
    friends {
      friends {
        friends {
          friends { id }
        }
      }
    }
  }
}
```

Set maximum depth based on schema and performance testing.

Enforce a maximum query depth (typically 5–10 levels) that rejects queries before execution:

javascript
const depthLimit = require('graphql-depth-limit');
const server = new ApolloServer({
  typeDefs, resolvers,
  validationRules: [depthLimit(7)]
});

#### Query Complexity Analysis

Assign costs to fields and reject queries above a threshold.

Example:

```text
user: 1
user.friends(first: N): 1 + N
search(query): 20
reportExport: 100
```
Depth limiting alone is insufficient — a shallow but wide query can be equally expensive. Assign a cost to each field and operation; reject queries exceeding a total cost threshold:

graphql
# Shallow but expensive: 1000 users x 100 posts each
{ users(limit: 1000) { posts(limit: 100) { title } } }


#### Field-Level Authorization

Resolvers must enforce access to fields, not just root queries.

Example:

```graphql
query {
  user(id: "123") {
    id
    email
    salary
    mfaRecoveryCodes
  }
}
```

Sensitive fields should require explicit authorization or not be present in the schema for that caller.

A single GraphQL query can request fields from multiple types. Field-level authorization must be enforced in each resolver:

javascript
const resolvers = {
  User: {
    email: (user, args, context) => {
      if (context.user.id !== user.id && !context.user.isAdmin) {
        return null;
      }
      return user.email;
    }
  }
};

Use a directive-based authorization framework (graphql-shield) to declare field-level rules declaratively, reducing the risk of accidental omission.

#### Introspection Control in Production

Introspection is useful for development and tooling but can disclose schema structure. In production, decide whether to disable it, restrict it to authenticated developers, or expose only to trusted environments.

Introspection gives clients a complete map of the API surface. Disable it in production:

javascript
const server = new ApolloServer({
  typeDefs, resolvers,
  introspection: process.env.NODE_ENV !== 'production'
});

### Rate Limiting for GraphQL

Request count alone is not enough because one GraphQL request can perform many expensive operations.

Use:

- Query cost analysis.
- Resolver-level timeouts.
- Batch query limits.
- Persisted queries for public clients.
- Maximum aliases.
- Maximum list sizes.
- Maximum input object depth.

Standard per-request rate limiting is insufficient because request complexity varies enormously. Use query cost analysis as the basis for cost-aware rate limiting — a client's rate limit is expressed in cost units per window, not request count.

Timeout enforcement terminates queries that exceed a maximum execution time, preventing long-running queries from monopolizing resources.

Batch query limits restrict the number of operations permitted in a single HTTP request.

### Common GraphQL Vulnerabilities

| Vulnerability | Example | Mitigation |
|---|---|---|
| DoS via complex queries | Deep nesting, large lists, expensive resolvers. | Depth, cost, timeout, and pagination limits. |
| Information disclosure | Introspection, verbose errors, hidden fields. | Restrict introspection and sanitize errors. |
| Authorization bypass | Resolver returns object without ownership check. | Authorize in every resolver and data loader. |
| Batch abuse | Many operations in one request. | Limit operation count and cost. |
| N+1 amplification | Query triggers excessive database calls. | DataLoader/batching plus complexity controls. |

DoS via complex queries: Addressed by depth limiting, complexity analysis, and timeout enforcement.

Information disclosure via introspection: Addressed by disabling introspection in production.

Authorization bypass in resolvers: Every resolver that accesses data must verify the requesting user's authorization for that specific data. A resolver that returns data based solely on query arguments without authorization checks is a BOLA/IDOR vulnerability in the GraphQL context.

## 13.8 SOAP and gRPC Security

### SOAP Security

SOAP is still common in enterprise, financial, government, and legacy integrations.

#### WS-Security Standards

WS-Security provides message-level mechanisms for SOAP, including message integrity, confidentiality, and security tokens.

Use cases:

- End-to-end message signing across intermediaries.
- Message encryption beyond transport TLS.
- Token-based identity in SOAP headers.

WS-Security defines mechanisms for XML signature, XML encryption, and token inclusion (SAML tokens, X.509 certificates) in SOAP headers. XML signature and encryption provide end-to-end security preserved through intermediaries that terminate TLS — unlike transport-level TLS which only protects adjacent hops.

#### XML Signature and Encryption

Security considerations:

- Validate signatures before trusting message content.
- Protect against signature wrapping attacks.
- Use secure canonicalization and parser settings.
- Keep keys and certificates managed and rotated.

#### SAML Token Integration

SOAP services may use SAML assertions for identity and authorization context.

Validate:

- Signature.
- Issuer.
- Audience.
- Expiry.
- Subject confirmation.
- Conditions.
- Replay protection.

#### XXE Prevention

XML External Entity attacks can read local files or trigger SSRF.

Disable:

- External entity resolution.
- DTD processing unless strictly required.
- External schema fetching.

Set parser limits for depth, entity expansion, and document size.

XXE Prevention in XML Parsing. SOAP services parse XML from clients. Disable DTD processing and external entity resolution in all XML parsers:

java
SAXParserFactory factory = SAXParserFactory.newInstance();
factory.setFeature("http://apache.org/xml/features/disallow-doctype-decl", true);
factory.setFeature("http://xml.org/sax/features/external-general-entities", false);
factory.setFeature("http://xml.org/sax/features/external-parameter-entities", false);

### gRPC Security

gRPC commonly uses HTTP/2 and Protocol Buffers.

#### TLS and mTLS

- Use TLS for transport encryption.
- Use mTLS for service-to-service identity where appropriate.
- Validate server names and certificate chains.
- Rotate certificates through service mesh or PKI automation.

TLS and mTLS for transport security. gRPC requires TLS in production. Mutual TLS (mTLS) authenticates both client and server, replacing or supplementing token-based authentication for service-to-service communication.

#### Token-Based Authentication

gRPC metadata can carry bearer tokens.

Validate tokens using the same rules as REST APIs:

- Signature.
- Expiry.
- Issuer.
- Audience.
- Scope.
- Tenant context.

#### Interceptors for Authorization

Use interceptors for consistent authentication, authorization, logging, and rate limiting.

Important: interceptors are good for cross-cutting controls, but method handlers may still need object-level checks.

Interceptors for authorization. gRPC interceptors provide a centralized point for authentication and authorization, analogous to middleware in REST frameworks:

go
func AuthInterceptor(ctx context.Context, req interface{},
    info *grpc.UnaryServerInfo, handler grpc.UnaryHandler) (interface{}, error) {

    token, err := extractTokenFromMetadata(ctx)
    if err != nil {
        return nil, status.Error(codes.Unauthenticated, "missing token")
    }
    claims, err := validateToken(token)
    if err != nil {
        return nil, status.Error(codes.Unauthenticated, "invalid token")
    }
    ctx = context.WithValue(ctx, claimsKey, claims)
    return handler(ctx, req)
}


#### Protobuf Validation

Protobuf schemas define types, but not all business rules.

Validate:

- Required business fields.
- String lengths.
- Enum values.
- Numeric ranges.
- Repeated field sizes.
- Resource ownership.
- State transitions.

Protobuf parsers silently ignore unknown fields by default. Explicitly validate messages against business constraints after deserialization — type safety does not substitute for business constraint validation.

## 13.9 Webhook Security

Webhooks invert the usual API model: an external system calls your endpoint. This creates authentication, replay, idempotency, and availability concerns.
Webhooks are HTTP callbacks: your application registers a URL with a third-party service and the service calls it when events occur.

### Common Webhook Vulnerabilities

| Vulnerability | Risk |
|---|---|
| Clear text communication | Payloads and secrets can be intercepted. |
| Missing verification of message and origin | Attackers can forge events. |
| Replay attacks | Old valid events can be submitted again. |
| Missing schema validation | Malformed payloads cause logic errors or injection. |
| Lack of expiry times | Old messages remain valid indefinitely. |
| SSRF via webhook URLs | Outbound webhook systems can be abused to reach internal services. |

Vulnerabilities

Missing verification of message and origin. Without signature verification, your endpoint cannot distinguish legitimate events from forged ones.

Replay attacks. A captured valid webhook payload can be replayed. Without timestamp validation, the same event can trigger duplicate processing.

SSRF via webhook URLs. If your application registers a user-controlled URL as a webhook target, an attacker can cause the third-party service to fetch arbitrary internal URLs.

Missing data and schema validation. Even legitimate payloads from trusted services must be validated — the service may change its response format, or a compromised sender can inject malicious payloads.

### Securing Incoming Webhooks

#### HTTPS Enforcement

Reject non-HTTPS webhook traffic except in local development tunnels.

#### HMAC Signature Verification

Typical pattern:

```http
X-Signature: sha256=<hex-hmac>
X-Timestamp: 1710000000
```

```pseudo
rawBody = request.rawBody
expected = hmac_sha256(webhookSecret, timestamp + "." + rawBody)
if !constantTimeEqual(expected, providedSignature):
    return 401
```

Important details:

- Sign the raw request body, not parsed JSON.
- Use constant-time comparison.
- Keep per-provider or per-tenant webhook secrets.
- Rotate webhook secrets.

HMAC Signature Verification

The standard mechanism: the sender computes an HMAC of the request body using a shared secret and includes it in a header. The receiver recomputes and compares.

python
import hmac
import hashlib

def verify_webhook_signature(
    payload: bytes,
    signature_header: str,
    secret: str
) -> bool:
    expected_sig = hmac.new(
        secret.encode(),
        payload,
        hashlib.sha256
    ).hexdigest()

    # Constant-time comparison prevents timing attacks
    return hmac.compare_digest(
        f"sha256={expected_sig}",
        signature_header
    )

Use hmac.compare_digest() — not == — for the comparison. Timing differences in string equality leak information about how much of the signature matched.

Compute the HMAC over raw request body bytes, not a parsed representation. Parse after verification.

#### Timestamp Validation and Replay Prevention

- Reject events outside a short tolerance window, such as five minutes.
- Store event IDs or signature hashes temporarily to prevent replay.
- Use idempotency keys when provided.

Include a timestamp in the signed payload. Reject payloads with timestamps outside an acceptable window (typically 5 minutes) regardless of signature validity. Maintain a cache of recently processed event IDs and reject duplicates within the replay window.

#### IP Allowlisting

IP allowlisting can help, but should not replace signature verification because provider IP ranges change and network controls can be misconfigured.

Many webhook senders publish source IP ranges. Configure network-level allowlists to accept webhook requests only from those ranges. This supplements, but does not replace, signature verification.

#### Request Timeout Limits

Webhook endpoints should respond quickly.

Recommended pattern:

1. Verify signature.
2. Validate schema.
3. Store event.
4. Return `2xx`.
5. Process asynchronously.

#### Payload Schema Validation

Validate event type and payload shape before processing.

### Webhook Endpoint Security

#### Authentication of Webhook Endpoints

Use HMAC signatures, mTLS, provider-specific signed headers, or strong shared secrets.

#### Idempotency Handling

Webhook providers often retry events.

```pseudo
if eventStore.exists(provider, event.id):
    return 200

eventStore.save(provider, event.id, payload)
queue.enqueue(event.id)
return 202
```
Idempotency handling. Webhook delivery systems use at-least-once delivery. Your handler must be idempotent — processing the same event multiple times must produce the same result as processing it once. Use the event ID to detect and skip duplicates.

#### Async Processing

Benefits:

- Faster response to provider.
- Better retry handling.
- Isolation from downstream failures.
- Easier monitoring.

Async processing. Acknowledge receipt immediately (HTTP 200) and process asynchronously. Long-running handlers trigger retries from the sender, producing duplicate deliveries.

python
@app.post("/webhooks/stripe")
async def stripe_webhook(request: Request, background_tasks: BackgroundTasks):
    payload = await request.body()

    if not verify_webhook_signature(payload, request.headers.get("Stripe-Signature"), STRIPE_SECRET):
        raise HTTPException(status_code=400)

    background_tasks.add_task(process_stripe_event, payload)
    return {"status": "accepted"}  # Return immediately

#### Error Handling and Retry Logic

- Return `2xx` only after accepting the event safely.
- Return `4xx` for invalid signatures or schemas.
- Return `5xx` for transient server failures.
- Avoid leaking validation internals.

Correct status codes for sender retry logic: Return 2xx to acknowledge receipt, 4xx for validation failures (sender should not retry), 5xx for processing errors (sender should retry). Never return 2xx for an invalid payload.

### Securing Outgoing Webhooks

When your system lets users register webhook URLs:

- Require HTTPS URLs.
- Prevent localhost, private network, link-local, and metadata service destinations.
- Validate DNS resolution and final redirect target.
- Set timeouts and response size limits.
- Sign outbound payloads.
- Support secret rotation.
- Provide delivery logs without exposing secrets.

## 13.10 API Gateway Security

An API gateway centralizes many controls, but it does not remove the need for secure application code.

An API gateway is the entry point for all API traffic, providing centralized authentication, authorization, rate limiting, logging, and traffic management.

### Gateway Functions

Common gateway responsibilities:

- TLS termination.
- Authentication and token validation.
- Coarse-grained authorization.
- Rate limiting and throttling.
- Request and response transformation.
- Routing.
- Logging and metrics.
- WAF integration.
- Bot and DDoS protection.

Security Features

WAF integration. Web Application Firewall rules at the gateway inspect request content for known attack signatures — SQL injection patterns, XSS payloads, path traversal sequences — as a defense-in-depth layer supplementing application-level validation.

Bot detection. Traffic pattern analysis, user agent inspection, and behavioral signals at the gateway identify and throttle automated traffic across all backend services simultaneously.

DDoS protection. Volumetric attack absorption at the CDN or cloud provider layer; connection rate limiting and protocol anomaly detection at the gateway.

### Centralized Authentication and Authorization

Gateway-level controls are useful for:

- Rejecting missing or invalid tokens early.
- Enforcing route-level scopes.
- Applying partner-specific policies.
- Protecting internal APIs behind mTLS.

Application services must still enforce:

- Object-level authorization.
- Field-level authorization.
- Business workflow authorization.
- Tenant isolation.

Centralized authentication and authorization. Token validation at the gateway means backend services do not each need to implement authentication logic. The gateway validates the token, extracts identity, and passes it to backends in a trusted header. This reduces duplicated security logic and the risk of inconsistent implementation.

### Request/Response Transformation Risks

Transformations can introduce security bugs.

Watch for:

- Removing security headers.
- Mapping external fields to internal privileged fields.
- Changing content types.
- Forwarding untrusted headers such as `X-User-Id`.
- Accidentally logging sensitive payloads.

Request and response transformation. The gateway can strip unexpected headers from requests and remove internal implementation details from error responses before they reach clients.

### Logging and Monitoring

Log enough to investigate abuse, but not secrets.

Good API security log fields:

- Request ID and trace ID.
- Authenticated subject and client ID.
- Tenant or organization ID.
- Endpoint and method.
- Status code.
- Authorization decision.
- Rate-limit decision.
- Latency.
- Source IP and user agent.
- Risk signals.

Redact:

- Authorization headers.
- API keys.
- Tokens.
- Passwords.
- OTPs.
- Payment data.
- Sensitive personal data.

Logging and monitoring. A single point for logging all API traffic: request metadata, response codes, authentication outcomes, and rate limit events.

### Configuration Security

Gateway configuration should be treated as production code.

Controls:

- Store config in version control.
- Review changes.
- Use environment-specific policy templates.
- Run automated config tests.
- Restrict admin access.
- Use MFA for admin consoles.
- Rotate certificates and upstream credentials.
- Keep default-deny routing.

Configuration Security

Secure default configurations. All endpoints require authentication by default. Any endpoint that bypasses authentication must be explicitly configured as public. Reverse defaults — public by default, protected by opt-in — are a common source of inadvertently exposed endpoints.

Certificate management. The gateway terminates client TLS and establishes TLS to backends. Use automated certificate management for public endpoints; never disable certificate validation for backend connections.

Secret management for backend services. Store gateway credentials for backend services in a dedicated secrets manager, not in configuration files. Use dynamic secret injection rather than embedding secrets in configuration.

Access control for gateway administration. Gateway administration grants the ability to bypass or modify all security controls — it requires the strongest authentication and authorization in the system, comprehensive audit logging, and restricted access.

#### Rate limiting and throttling
Gateway-level rate limiting protects backend services before requests reach them.

## 13.11 Third-Party API Security

Third-party APIs extend your trust boundary. Fail closed where possible and design for dependency failure.

### CORS

CORS controls which browser origins can read responses. It does not stop non-browser clients.

Secure CORS rules:

- Allow exact origins, not broad wildcards.
- Do not use wildcard origins with credentials.
- Keep allowed methods minimal.
- Keep allowed headers minimal.
- Handle preflight requests consistently.
- Do not dynamically reflect any `Origin` without validation.

CORS controls which origins may make cross-origin requests from browser JavaScript. Server-side clients are not subject to CORS.

### Third-Party Authentication and Authorization

When accepting third-party tokens:

- Verify token signature with trusted keys.
- Validate issuer and audience.
- Validate expiry.
- Confirm scopes or permissions.
- Map external identity to internal identity carefully.
- Avoid account takeover through unverified email claims.

When storing third-party tokens:

- Encrypt at rest.
- Limit access by service role.
- Rotate when supported.
- Revoke on disconnect.
- Avoid logging tokens.

Validating third-party tokens. When accepting tokens from a third-party identity provider (Auth0, Okta, Google), validate the token fully: signature verification against the provider's JWKS, issuer validation, audience validation, and expiry. The presence of a token from a recognized provider is not sufficient — validate that it was issued for your specific application.

Scope verification. Verify the token's scopes match what is required for the requested operation. A token with scope=profile should not grant access to endpoints requiring scope=admin.

Token storage security. In browser applications: prefer memory storage for short-lived access tokens over localStorage (accessible to any script on the page). Use HttpOnly, Secure, SameSite=Strict cookies for session management.

Proper origin validation. Access-Control-Allow-Origin must be a specific origin or a dynamically validated origin from an allowlist — never * for endpoints serving authenticated data.

Credential handling. Access-Control-Allow-Credentials: true must only be combined with a specific (non-wildcard) Access-Control-Allow-Origin header. The browser rejects * combined with credentials, but the misconfiguration still represents a design flaw.

Avoiding wildcard origins with credentials. Validate the Origin header against an explicit allowlist and reflect it conditionally:

python
ALLOWED_ORIGINS = {
    "https://app.example.com",
    "https://dashboard.example.com"
}

@app.middleware("http")
async def cors_middleware(request: Request, call_next):
    origin = request.headers.get("Origin")
    response = await call_next(request)

    if origin in ALLOWED_ORIGINS:
        response.headers["Access-Control-Allow-Origin"] = origin
        response.headers["Access-Control-Allow-Credentials"] = "true"
        response.headers["Vary"] = "Origin"  # Vary by Origin for correct caching

    return response

### Data Validation from External APIs

Never trust external API responses, even from reputable providers.

Validate:

- Schema shape.
- Enum values.
- URL destinations.
- Numeric bounds.
- Currency and locale assumptions.
- Character encoding.
- Expected signatures, when provided.

Never trust external API responses. External APIs can be compromised, can change response formats, and can return unexpected values. Treat external API responses as untrusted input.

Schema validation of responses. Validate external API responses against the expected schema before using the data. A response that does not match the expected structure is an error condition, not something to silently process.

Error handling for external failures. External APIs can fail through network timeout, service unavailability, rate limiting, or unexpected response format. Handle each failure mode explicitly. Never allow external API failures to propagate unhandled exceptions to users.

Timeout and circuit breaker patterns. Every external API call must have a defined timeout. Without timeouts, a slow external API exhausts your application's connection pool, causing cascading failures. The circuit breaker pattern prevents repeated calls to a consistently failing service, allowing graceful degradation.

### Error Handling for External Failures

Do not let dependency failures cascade.

Use:

- Timeouts.
- Retries with exponential backoff and jitter.
- Circuit breakers.
- Bulkheads.
- Fallback responses.
- Dead-letter queues.
- Clear operational alerts.

### Timeout and Circuit Breaker Patterns

Example policy:

```text
Connection timeout: 500 ms
Read timeout: 2 seconds
Total retry attempts: 2
Backoff: exponential with jitter
Circuit opens after: 50% failures over 20 requests
Half-open probe after: 30 seconds
```

Tune these values per operation criticality and provider reliability.

## 13.12 API Security Testing

API security testing should combine automated tests, manual abuse-case testing, and production monitoring.

### Security Testing Approaches

#### API Fuzzing

Fuzzing sends unexpected inputs to find crashes, validation gaps, injection issues, and inconsistent behavior.

Good fuzz targets:

- OpenAPI request schemas.
- GraphQL queries and variables.
- File uploads.
- Numeric boundaries.
- String encoding.
- Date parsing.
- Filter/sort parameters.
- Batch endpoints.

API fuzzing sends malformed, unexpected, or boundary-value inputs to API endpoints to identify crashes and unexpected behavior. Unlike web application fuzzing, API fuzzing targets structured inputs (JSON fields, path parameters, header values) with type-aware mutations.

Schemathesis is particularly effective — it generates test cases from the OpenAPI specification, testing all documented endpoints with both conforming and non-conforming inputs, looking for 5xx responses and schema violations:

bash
schemathesis run https://api.example.com/openapi.json \
  --auth "Bearer $TOKEN" \
  --checks all

#### Authentication Testing

Test:

- Missing token.
- Expired token.
- Token signed by wrong issuer.
- Token with wrong audience.
- Token with insufficient scope.
- Token using disallowed algorithm.
- Revoked token.
- Malformed token.

Authentication testing verifies that: unauthenticated requests are rejected consistently, tokens are properly validated (expiry, signature, issuer, audience, algorithm), token reuse after logout is prevented, and elevated-privilege tokens cannot be forged.

#### Authorization Testing

Test matrix:

| Test Type | Example |
|---|---|
| Horizontal access | User A reads User B's invoice. |
| Vertical access | Normal user calls admin operation. |
| Tenant isolation | User from tenant A accesses tenant B resource. |
| Field-level read | User reads restricted fields. |
| Field-level write | User updates protected fields. |
| Batch access | User requests mixed authorized and unauthorized IDs. |
| Search access | User retrieves unauthorized resource through filters. |

Authorization testing is the most critical and most manually intensive testing category. For every endpoint, test with: a completely unauthenticated request; a request authenticated without the required role; a request in the correct role attempting to access another user's data (BOLA/IDOR testing). For BOLA, systematically test object IDs belonging to other accounts and verify the response is consistently 404 or 403.

#### Business Logic Testing

Examples:

- Can a coupon be reused?
- Can a user reset someone else's MFA?
- Can a refund exceed limits?
- Can an invite be accepted after expiry?
- Can a workflow step be skipped?
- Can a user create unlimited trial accounts?

Business logic testing for APIs requires understanding the application domain. For each business rule enforced by the API (discount limits, quantity constraints, workflow sequences), test what happens when the rule is violated. Module 11 covers API business logic testing in detail.

#### Rate Limit Testing

Verify:

- Limits trigger at expected thresholds.
- Limits apply per correct identity dimension.
- Expensive endpoints have stricter limits.
- Distributed deployments share counters correctly.
- `429` responses include useful retry information.
- Login and reset endpoints resist brute force.

Rate limit testing verifies that limits are enforced at the claimed thresholds, cannot be bypassed by rotating IP addresses or user agents, and that rate-limited responses return HTTP 429 with appropriate headers.

#### Injection testing
Injection testing applies SQL injection, NoSQL operator injection, and command injection payloads to all API string inputs, including request body fields, path parameters, and query parameters.


#### OpenAPI-driven testing
OpenAPI-driven testing uses the API specification as a test oracle, ensuring comprehensive endpoint coverage. Every documented endpoint is tested, not just those discovered through manual exploration. Schemathesis, Dredd, and similar tools implement this pattern.

### Continuous Security

#### API Security in CI/CD

Include:

- Unit tests for authorization policies.
- Contract tests from OpenAPI/GraphQL/gRPC schemas.
- Static analysis for injection patterns and secret leakage.
- Dependency scanning.
- Container and infrastructure scanning.
- Dynamic API scans in test environments.
- Fuzzing for critical endpoints.

#### Automated Security Scanning

Automated tools help with known classes, but they often miss business logic and object-level authorization. Treat scanner output as one signal, not a complete answer.

#### API Contract Testing

Contract tests prevent drift between documentation and implementation.

Test that:

- Required authentication is documented and enforced.
- Request schemas match runtime validation.
- Response schemas do not expose unexpected fields.
- Error responses follow standard format.
- Deprecated endpoints are tracked.

#### Security Regression Testing

For every API security bug, add a regression test.

Example:

```pseudo
test "user cannot read another user's invoice" {
    alice = createUser()
    bob = createUser()
    invoice = createInvoice(owner=bob)

    response = api.as(alice).get("/invoices/" + invoice.id)

    assert response.status in [403, 404]
}
```

## Developer Code Review Checklist

Use this checklist during pull requests for API changes.

### Authentication

- [ ] Does the endpoint require authentication unless intentionally public?
- [ ] Are tokens or API keys validated centrally?
- [ ] Are issuer, audience, expiry, signature, and algorithm validated?
- [ ] Are secrets absent from URLs, logs, and error messages?

### Authorization

- [ ] Is function-level authorization enforced?
- [ ] Is object-level authorization enforced for every resource ID?
- [ ] Are tenant and organization boundaries checked?
- [ ] Are field-level read/write restrictions enforced?
- [ ] Are batch and search endpoints filtered safely?

### Validation

- [ ] Is request schema validation present?
- [ ] Are content type and body size enforced?
- [ ] Are sort, filter, and pagination parameters allowlisted?
- [ ] Are database and shell operations parameterized or safely wrapped?

### Resource Protection

- [ ] Are rate limits and quotas appropriate for endpoint cost?
- [ ] Are timeouts and response size limits set?
- [ ] Are expensive operations asynchronous or capped?
- [ ] Are login, reset, OTP, and token endpoints protected from brute force?

### Response Safety

- [ ] Are response serializers allowlisted?
- [ ] Are sensitive fields omitted?
- [ ] Are errors generic but useful?
- [ ] Are security headers present where applicable?
- [ ] Is sensitive data protected from caching?

### Integrations

- [ ] Are third-party responses validated?
- [ ] Are outbound calls protected by timeouts and circuit breakers?
- [ ] Are webhook signatures verified?
- [ ] Are replay and idempotency controls implemented?
- [ ] Are SSRF protections in place for user-provided URLs?

### Operations

- [ ] Is the endpoint included in API inventory?
- [ ] Is ownership documented?
- [ ] Are logs useful and redacted?
- [ ] Are alerts or dashboards needed?
- [ ] Are deprecation/versioning concerns addressed?

## Key Takeaways

- Authentication proves identity; authorization must still be enforced for each function, object, field, and workflow.
- BOLA and BFLA are design and implementation failures, not just configuration issues.
- Tokens must be verified, not merely decoded.
- Rate limiting should reflect cost, identity, endpoint risk, and business abuse potential.
- Input validation should be schema-based, strict, and paired with safe database and command APIs.
- GraphQL needs cost controls because one request can represent a large operation graph.
- Webhooks require signature verification, replay protection, idempotency, and async processing.
- API gateways help centralize controls, but service code still needs fine-grained authorization.
- Third-party APIs are untrusted inputs and unreliable dependencies.
- Security tests should become regression tests.

## Appendix: Example Secure Endpoint Skeleton

```pseudo
route PATCH /api/v1/projects/{projectId}/members/{memberId}

function updateProjectMember(request):
    user = auth.requireUser(request)
    auth.requireScope(request.token, "project.member:write")

    validateContentType(request, "application/json")
    validateBodySize(request, maxBytes=16_384)
    body = validateSchema(request.body, UpdateMemberSchema)

    project = projectRepository.find(request.path.projectId)
    if project is null:
        return 404

    member = memberRepository.find(request.path.memberId)
    if member is null or member.projectId != project.id:
        return 404

    authorize(user, "project.member.update", project)
    authorize(user, "member.update", member)

    update = pick(body, ["role", "displayName"])

    if update.role == "owner":
        authorize(user, "project.owner.assign", project)

    updated = memberService.update(member, update)

    audit.log("project_member_updated", {
        actor: user.id,
        project: project.id,
        member: member.id,
        changedFields: keys(update)
    })

    return serializeMemberFor(user, updated)
```
