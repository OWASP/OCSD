# Supplementary Reading Material - Security Architecture and Threat Modelling

This material examines the topics that a developer needs to know about security architecture and threat modelling.

## Security Architecture
We will not be diving into the history of security architecture or explaining about Zachman Framework or TOGAF Framework.

* So, what is security architecture?   
The Secure Architecture (SA) practice focuses on the security linked to components and technology you deal with during the architectural design of your software. Secure Architecture Design looks at the selection and composition of components that form the foundation of your solution, focusing on its security properties. Technology Management looks at the security of supporting technologies used during development, deployment and operations, such as development stacks and tooling, deployment tooling, and operating systems and tooling. <<https://owaspsamm.org/model/design/secure-architecture/>>

## Software Development Lifecycle (SDLC)
SDLC stands for software development lifecycle. It’s a ubiquitous framework for managing an entire software project from start to finish, ensuring consistency, efficiency, and quality. Typically, the SDLC has several key stages, including requirements gathering and analysis, system design, coding, testing, deployment, and maintenance and support. Ideally, this is a circular process that eventually feeds back into the initial planning stage, ensuring that the software continues to evolve and improve.

The SDLC development helps development teams efficiently build high-quality software at a lower cost while ensuring that it meets stakeholder requirements and adheres to their organization’s standards for quality, security, and compliance. Today's teams are increasingly leveraging AI tools to enhance efficiency and quality throughout the entire SDLC. <<https://github.com/resources/articles/what-is-sdlc>>

Typical SDLC activities include requirements engineering, architecture and design, implementation, testing, deployment, operations, maintenance, and decommissioning. For developers, the key insight is that every phase creates security-relevant decisions. Requirements define what must be protected. Design determines trust boundaries and control points. Coding determines whether safeguards are implemented correctly. Testing checks whether defects remain exploitable. Operations determines whether the software stays secure after release through logging, patching, monitoring, and configuration management. Security failures often arise not because teams skipped coding standards, but because security was omitted in earlier lifecycle activities such as architecture, trust modeling, or dependency decisions. OWASP’s work on insecure design emphasizes that many serious weaknesses originate before code is written.

Common SDLC models include Waterfall, V-Model, Iterative, Spiral, Agile, and DevOps/DevSecOps-style continuous delivery. No single model is automatically secure. What matters is whether security work is integrated into the cadence of the chosen model. In sequential models, security often appears as formal reviews and phase gates. In Agile, security should appear in backlog refinement, architecture decisions, sprint planning, code review, automated testing, and release criteria. In DevSecOps, security controls are automated where possible, such as static analysis, dependency checking, secrets detection, infrastructure scanning, and policy enforcement in CI/CD. NIST RMF’s lifecycle-oriented view supports this idea: risk management must be continuous, not episodic

## Secure-by-Design Principles
Secure by design means building systems so that security is part of the default architecture, not an optional add-on. This includes minimizing attack surface, validating inputs, using safe defaults, separating trust zones, protecting secrets properly, designing for patchability, and choosing libraries and protocols with strong security properties. OWASP identifies insecure design as a major category of risk and calls for more use of threat modeling, secure design patterns, and reference architectures. For developers, secure design starts before implementation and continues through code structure, API contracts, deployment assumptions, and operational behavior.

Systems fail. The question is whether they fail in a way that increases risk. Fail securely means that when an error occurs, access is denied or the action is safely constrained. Fail safely adds the idea that the system should not create physical or operational harm when it enters an abnormal state. For example, an authorization error should not result in unrestricted access, and a service outage should not expose sensitive data. Error handling, timeouts, retries, and fallback modes are design decisions with security consequences.

Defense in depth means using multiple, independent or semi-independent layers of protection so that failure of one control does not immediately lead to compromise. Examples include authentication plus authorization, encryption plus key management controls, application validation plus database controls, and logging plus alerting plus incident response. Developers should not rely on a single line of defense such as a frontend validation check or a network boundary. Layered controls are especially important in distributed systems and APIs where trust assumptions can fail in more than one place.

Segregation of duties reduces risk by ensuring that no single actor can perform a sensitive action end to end without oversight or separation. In software systems this may include requiring separate roles for approving deployments, rotating production secrets, administering databases, or authorizing high-value transactions. It also applies inside applications: an operation that creates, approves, and executes a payment should not be possible through one role unless explicitly justified and controlled.

Least privilege means giving users, services, workloads, and tools only the permissions they need, only for as long as they need them. For developers, this affects IAM roles, service accounts, API scopes, database permissions, container privileges, and feature access inside the application. Least privilege reduces blast radius. A bug in a low-privilege component should not become full-system compromise because the component was over-permissioned.

Economy of mechanism means keeping security mechanisms as simple and small as practical. Complexity increases misunderstanding, misconfiguration, and hidden failure modes. Developers should prefer simple trust models, narrow interfaces, well-understood protocols, and fewer special cases. Small, clear components are easier to review, test, and harden than sprawling logic with ambiguous behavior.

Complete mediation means every access to every protected object should be checked against current policy, not assumed safe because of prior checks. This is closely aligned with zero trust thinking: never rely solely on network location or previous state as proof of trust. Every request should be authenticated, authorized, validated, and logged as appropriate. In modern architectures, this means treating every API call, service request, token exchange, and administrative action as something that must be verified in context.

## Common Design Flaws
The OWASP Top 10 is a widely used awareness document for developers that summarizes major web application security risks. OWASP states that the current released version is OWASP Top 10: 2025, and earlier editions remain available for reference. The Top 10 is not a complete security model, but it is an effective teaching and prioritization tool. It helps teams recognize recurring risk patterns such as broken access control, cryptographic failures, injection, insecure design, security misconfiguration, vulnerable components, identification and authentication failures, software and data integrity failures, logging and monitoring failures, and server-side request forgery. Developers should use it as a starting point for awareness, not as the only checklist for secure design.

A useful lesson from OWASP is that many categories are not just “bugs.” They are symptoms of deeper design weaknesses:
    * missing trust boundaries,
    * weak authorization models,
    * poor dependency governance,
    * unsafe defaults,
    * absent logging and response hooks,
    * and inadequate threat modeling.

## Threat Modeling and Risk Management
Threat modeling is a structured way to think about what you are building, what can go wrong, what matters most, and what you should do about it. It helps teams identify assets, trust boundaries, entry points, misuse cases, abuse paths, and likely attacker actions before systems are deployed. OWASP’s guidance on insecure design specifically calls for more threat modeling during pre-code activities.

Threat modeling is not only for security specialists. Developers, architects, testers, product owners, and operations teams all benefit from it because design decisions create security consequences across the whole lifecycle. A threat model makes hidden assumptions visible. It also gives teams a shared vocabulary for discussing risk without waiting for a penetration test or incident.

Threat modeling supports secure development by helping teams:
    * identify security requirements earlier,
    * find design flaws before code is expensive to change,
    * prioritize testing and review effort,
    * choose mitigations based on actual attack paths,
    * justify residual risk decisions,
    * and document why certain controls were selected.

Understanding threat modeling-related terms
A few terms matter:

    * Asset: something of value that must be protected, such as credentials, personal data, transactions, secrets, models, logs, service availability, or safety functions.
    * Threat: a potential cause of harm, such as spoofing, data disclosure, privilege abuse, or service disruption.
    * Vulnerability: a weakness that can be exploited by a threat.
    * Attack vector: the path or method used to exploit a vulnerability.
    * Risk: the combination of likelihood and impact associated with a threat exploiting a vulnerability against an asset.
    * Control / mitigation: a safeguard that prevents, detects, reduces, transfers, or compensates for risk.
    * Residual risk: the risk that remains after controls are applied.

A developer-friendly rule is:\
assets tell you what matters, threats tell you what can happen, vulnerabilities tell you why it can happen, and risk tells you how much it matters.

Commonly used threat model methods
Dataflow Diagrams (DFDs)\
Dataflow Diagrams (DFDs) are a common way to represent processes, data stores, external entities, and the flows between them. They help developers visualize trust boundaries and data movement. A good DFD is often the backbone of a useful threat model because many threats become obvious once you see where data enters, crosses boundaries, is stored, or leaves the system.

Threat modeling approaches and the Four Questions\
    A practical approach is to ask four questions:
    * What are we building?
    * What can go wrong?
    * What are we going to do about it?
    * Did we do a good enough job?

This keeps the exercise grounded in engineering decisions rather than abstract brainstorming.

Threat identification aids
Different methods help teams identify and reason about threats:

STRIDE: categorizes threats as Spoofing, Tampering, Repudiation, Information Disclosure, Denial of Service, and Elevation of Privilege. Good for system and software threats.

DREAD: historically used to score threats by factors such as damage and reproducibility. It is more a ranking aid than a modeling method.

LINDDUN: privacy-focused and useful when systems process personal data. It emphasizes privacy harms that may not be captured by purely security-oriented models.

EoP / Cornucopia: ways to stimulate thinking about abuse paths and controls. Cornucopia in particular is often used as a practical card-based aid for secure design discussion.

Threat libraries\
Threat libraries provide example attack patterns, weaknesses, and adversary behaviors. CAPEC is a well-known example for attack patterns and is useful when teams want concrete examples of how an attacker could exploit a design.

Threat modeling tools\
Threat modeling tools help teams create diagrams, manage threats, map mitigations, and keep models current as systems evolve. The best tool is the one that fits the team’s workflow and is actually used. A simple, maintained model is more valuable than a sophisticated model that no one updates.

Threat models as sources of security requirements
A threat model is not just a diagram or workshop output. It is a source of security requirements. Once a threat is identified and accepted as relevant, the chosen remediation should become a requirement, design constraint, backlog item, test objective, deployment control, or operational control.

Accept inputs from threat models for secure development\
Developers should treat threat-model outputs as engineering inputs. Examples:
      * “All admin actions must require strong authentication.”
      * “Sensitive fields must be encrypted at rest.”
      * “Every cross-tenant query must enforce tenant context.”
      * “Privileged APIs must be rate-limited and audited.”
      * “The service must reject unsigned or tampered webhook requests.”

Selected remediations are system requirements\
A selected mitigation is not complete until it has an owner, an implementation plan, and a verification method. If the threat model says “protect secrets,” the actual requirement must say how: for example, “application secrets shall be stored only in the approved secrets manager and never in source control, container images, or client-side code.”

Inputs for secure coding and compliance\
Threat models also provide inputs to coding standards and compliance activities. PCI DSS, for example, expects secure development practices and requires security vulnerabilities to be identified and addressed. Teams in regulated environments should map threat-model findings to the specific controls, contractual obligations, and legal requirements that apply to the system. This prevents the model from becoming isolated from the real governance environment.

Risk Management Concepts\
Available options for addressing identified risks\
Once risks are identified, common treatment options are:
      * Mitigate: reduce likelihood or impact with controls.
      * Avoid: remove the risky feature, path, dependency, or architecture choice.
      * Transfer: shift some financial or operational consequence, for example through contracts or insurance.
      * Accept: knowingly retain the risk based on context and appetite.

Acceptance should never mean “ignored.” It means documented, understood, and owned.

Risk-based decisions for acceptance of residual risks\
No real system has zero risk. After controls are applied, teams must decide whether the remaining risk is acceptable. That decision should consider impact, likelihood, exploitability, business value, safety concerns, privacy implications, customer expectations, and regulatory obligations. NIST RMF describes risk management as structured, repeatable, and measurable, and ties authorization decisions to ongoing risk understanding and continuous monitoring.

Certification and Accreditation (C&A) for software systems\
In formal environments, software systems may require authorization before production use. Historically this has often been called Certification and Accreditation (C&A). Modern NIST guidance emphasizes an authorization approach within the RMF that includes categorization, control selection, implementation, assessment, authorization, and continuous monitoring. The important lesson for developers is that security approval is not only about passing a test once. It is about showing that risks are understood, controls are implemented, evidence exists, and the system remains trustworthy over time.

## Recommended developer takeaways

By the end of this module, a developer should be able to:
      * explain why security belongs in every SDLC phase,
      * apply secure-by-design principles during architecture and implementation,
      * recognize common design flaws using OWASP guidance,
      * participate meaningfully in threat modeling,
      * convert threats into security requirements,
      * and support risk-based decisions with clear technical evidence.
