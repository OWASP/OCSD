# Supplementary Reading Materials - Secure Development Environment

# 16.1 Context and Threat Model

## Why Developer Workstations Are High-Value Targets

Developer machines are different from standard office workstations. They often have broader access, more powerful tooling, and fewer restrictions because developers need flexibility to build, test, debug, and deploy systems. This flexibility creates risk.

Attackers may target developer workstations to:

- Steal source code or intellectual property.
- Obtain cloud credentials, API keys, SSH keys, or tokens.
- Modify source code or build pipelines.
- Insert malicious dependencies or backdoors.
- Access internal systems through VPN, SSH, SSO, or browser sessions.
- Exfiltrate sensitive data used for testing or debugging.
- Abuse package managers, IDE extensions, terminal tools, or local scripts.

A developer workstation is not just another endpoint. It sits at the intersection of source code, production credentials, cloud access, CI/CD pipelines, and sensitive business logic. Compromising a single developer machine can give an attacker a foothold that bypasses perimeter defenses entirely.

The 2020 SolarWinds supply chain attack is a landmark example: attackers compromised the build environment of a trusted software vendor and injected malicious code into signed updates delivered to thousands of organizations. The entry point was not a misconfigured server — it was a developer and build environment.

## Common Threat Scenarios

### 1. Credential Theft

A developer stores a cloud API key in a local `.env` file. Malware, a malicious plugin, or an accidentally committed file exposes the key. The attacker uses it to access cloud resources or internal APIs.

### 2. Malicious Dependency or Package

A developer installs an open-source package from a public registry. The package contains a post-install script that steals environment variables, SSH keys, or package registry tokens.

### 3. Compromised IDE Extension

A popular extension is hijacked or replaced by a malicious version. It reads files from the workspace and sends source code or secrets to an external server.

### 4. Unsafe Online Tool Usage

A developer pastes logs, stack traces, JWTs, customer records, or proprietary code into a free online formatter, AI tool, code converter, or debugging service. Sensitive data leaves organizational control.

### 5. Insecure Remote Work

A developer works from home or a public location using weak Wi-Fi, unmanaged personal devices, or insecure remote access. An attacker intercepts traffic, steals sessions, or compromises the machine.

### 6. Container Escape or Unsafe Local Runtime

A developer runs unknown Docker images with privileged mode or host-mounted sensitive directories. The container reads files from the host, modifies local tools, or accesses credentials.

## Assets to Protect

Developers should treat the following as sensitive:

- Source code and internal repositories.
- Build artifacts and deployment packages.
- API tokens, credentials, passwords, SSH keys, certificates, and private keys.
- Cloud profiles and kubeconfig files.
- Database dumps, sample data, logs, and debugging output.
- Customer, employee, or business data.
- Browser sessions and SSO cookies.
- Internal architecture diagrams, tickets, runbooks, and incident details.
- CI/CD configuration and deployment scripts.

## The Developer Threat Model
Threat modelling is the process of identifying what can go wrong, who might cause it, and how to prioritize defenses. For a developer workstation, the relevant threats include:

| Threat | Example |
|---|---|
| **Malicious packages** | A typosquatted npm package exfiltrates environment variables |
| **Credential theft** | SSH private keys or API tokens stolen via malware |
| **Supply chain compromise** | A dependency update introduces a backdoor |
| **Insider threat** | A disgruntled developer intentionally leaks source code |
| **Phishing** | A developer clicks a link that installs a keylogger |
| **IDE plugin compromise** | A malicious VS Code extension exfiltrates clipboard data |

# 16.2 Core Principles for Secure Workstations

Secure workstation practices are based on a few core principles. These principles should guide everyday decisions, not just formal security reviews.

## Principle 1: Least Privilege

Use only the access needed for the task. Avoid using administrator, root, production, or highly privileged accounts for routine development.

Practical examples:

- Use a standard user account for daily work.
- Use privilege escalation only when required.
- Keep production credentials separate from development credentials.
- Use short-lived access tokens when possible.
- Avoid giving local tools broad access to cloud accounts or internal systems.

Your workstation user account should have the minimum permissions needed to do your job. Running as a local administrator (Windows) or using `sudo` habitually (Linux/macOS) means that any process you inadvertently execute — malicious or not — inherits those elevated privileges.

- Use a standard user account for daily work
- Elevate only when explicitly required and with awareness of what you are doing
- Never run your IDE, browser, or email client as root or administrator

## Principle 2: Separation of Duties and Environments

Development, testing, staging, and production should be separated. Workstations should not become informal production administration consoles unless explicitly approved.

Practical examples:

- Use separate profiles for development and production cloud accounts.
- Avoid storing production data locally.
- Use separate browser profiles for personal and work accounts.
- Do not mix personal projects and company code in the same environment.

Keep work and personal use separate. Using a work laptop for personal browsing, personal email, or gaming introduces threat vectors that your organization has no visibility into or control over.

- Use separate browser profiles for work and personal use at minimum
- Ideally, use separate physical devices or properly isolated VMs

## Principle 3: Defense in Depth

No single control is enough. Security should rely on multiple layers such as device encryption, endpoint protection, patching, MFA, secrets scanning, logging, and access control.

If one control fails, another should reduce impact.

No single control is sufficient. Layering multiple controls means that if one fails, others remain.

- Disk encryption + strong password + screen lock + endpoint security = layered protection
- Do not rely solely on the corporate firewall or VPN to protect you

## Principle 4: Secure Defaults

Choose configurations that are safe by default. Developers should not have to remember complex manual steps every time.

Examples:

- Disk encryption enabled by default.
- Automatic updates enabled.
- Secrets scanning integrated into Git workflows.
- MFA required for code hosting, cloud accounts, and package registries.
- Standard approved IDE and plugin baselines.

**Keep Systems Updated**
The majority of successful attacks exploit known, patched vulnerabilities. Delaying updates is a choice to remain vulnerable.

- Enable automatic OS updates
- Regularly update developer tools, runtimes, and package managers
- Treat update fatigue as a risk to be managed, not a reason to skip patches

## Principle 5: Minimize Local Sensitive Data

The safest local secret or dataset is the one that does not exist. Avoid downloading production data, long-lived secrets, or unnecessary logs.

When local sensitive data is unavoidable:

- Store it encrypted.
- Limit retention.
- Remove it after use.
- Do not sync it to personal cloud storage.
- Do not paste it into external tools.

## Principle 6: Verify Before Trusting

Developers often install tools quickly. Treat every local binary, script, extension, container image, package, or online service as something that needs trust evaluation.

Ask:

- Who published it?
- Is it actively maintained?
- Does it request unnecessary permissions?
- Is it widely used and reviewed?
- Could it access sensitive files or tokens?
- Is there an approved internal alternative?

### Workstation Hardening Checklist

| Control | Why It Matters |
|---|---|
| Full disk encryption (FileVault, BitLocker, LUKS) | Protects data if device is lost or stolen |
| Screen lock after inactivity (≤5 minutes) | Prevents physical access when unattended |
| Strong local account password or PIN | First line of defence against physical access |
| Endpoint security / EDR agent | Detects and blocks malware, suspicious behaviour |
| Firewall enabled (host-based) | Limits inbound connections to the workstation |
| Automatic OS and software updates | Reduces exposure to known vulnerabilities |
| Removal of unnecessary software | Reduces attack surface |
| Secure boot enabled | Prevents boot-level rootkits |

# 16.3 Policies, Roles, and Responsibilities

Security is a shared responsibility. Developers, security teams, IT, engineering managers, and platform teams each play a role.
Security policies are not bureaucratic obstacles — they are documented decisions made by your organization about acceptable risk. When you understand the intent behind a policy, you are better positioned to follow it in spirit as well as in letter, and to raise concerns when a policy is impractical.

## Developer Responsibilities

Developers are responsible for:

- Keeping workstations patched and compliant (if using personal devices)
- Protecting credentials, tokens, keys, and sensitive data.
- Using approved tools and services where required.
- Reporting lost devices, suspected compromise, leaked secrets, or unusual behavior.
- Avoiding unapproved storage or sharing of sensitive information.
- Reviewing plugins, dependencies, scripts, and containers before use.
- Following secure coding and secure development lifecycle requirements.

responsibilities also include:

- Writing code that implements security controls, not just functional requirements
- Raising security concerns during design and code review, not just after a pen test
- Reporting suspected incidents or anomalies (phishing attempts, suspicious processes, unexpected credential prompts)
- Protecting credentials and secrets you are entrusted with
- Completing required security training and staying current with emerging threats

## Engineering Manager Responsibilities

Managers and technical leads should:

- Ensure developers understand workstation security expectations.
- Avoid pressuring teams to bypass security controls for speed.
- Support time for patching, upgrades, and security training.
- Ensure access is reviewed when developers change roles or projects.
- Promote security champions and secure development habits.

## Security Team Responsibilities

Security teams should:

- Define practical policies that developers can follow.
- Provide approved tool lists and secure configuration guidance.
- Offer secrets scanning, dependency scanning, and endpoint monitoring.
- Support incident response for workstation compromise.
- Review high-risk tools, plugins, SaaS services, and AI-assisted development platforms.

## IT / Endpoint Management Responsibilities

IT or endpoint teams should:

- Provision devices securely.
- Enforce disk encryption, patching, endpoint protection, and device inventory.
- Manage device recovery, remote wipe, and lost-device response.
- Provide secure remote access mechanisms.
- Support developer needs without encouraging unmanaged workarounds.

## Policy Topics Developers Should Know

Developers should be familiar with organizational policies for:

- Acceptable use of company devices. 
Defines what you may and may not do with company-issued equipment and network access. Commonly restricts: personal use of work devices, installation of unauthorized software, connecting to public Wi-Fi without a VPN.
- Bring-your-own-device rules.
- Software Installation Policy - Specifies how new software (including development tools, plugins, and packages) must be vetted and approved. In mature organizations, this involves a software inventory and approval workflow.
- Approved IDEs, plugins, and package managers.
- Cloud service usage.
- AI and online tool usage.
- Secrets management.
- Production access. Governs access to production systems, databases, and secrets. Developers typically should not have standing access to production environments — access should be time-limited and logged.
- Data classification and handling. - Defines categories of data sensitivity (e.g. Public, Internal, Confidential, Restricted) and the controls required for each. As a developer, you need to know:\
    What classification applies to the data your application processes\
    Whether that data can appear in logs, test data, or local development environments\
    What happens if classified data is accidentally committed to a repository\

- Remote work and home office security.
- Incident reporting.

## Reporting Expectations

Report immediately when:

- A laptop, phone, hardware token, or storage device is lost or stolen.
- A secret may have been committed, shared, logged, or exposed.
- A suspicious extension, dependency, script, or binary was installed.
- Antivirus or endpoint protection flags an issue.
- You notice unexpected logins, MFA prompts, or cloud activity.
- You accidentally pasted sensitive data into an external service.

Fast reporting reduces damage. Security teams can rotate credentials, revoke sessions, isolate devices, and investigate activity.

# 16.4 Use of Local Tools, IDEs, and Plugins

Local development tools are powerful. They can read code, execute commands, access terminals, index files, inspect environment variables, and communicate with external services.

## IDE and Editor Security

Common IDEs and editors support extensions that can significantly change behavior. Extensions may access:

- Workspace files.
- Terminal sessions.
- Git repositories.
- Environment variables.
- Language servers.
- Debug sessions.
- Clipboard contents.
- Network resources.

### The IDE as an Attack Surface

Your Integrated Development Environment (IDE) has deep access to your workstation: it reads and writes files, executes code, communicates with external services, and often handles credentials. A malicious or compromised plugin can silently exfiltrate source code, steal tokens from your keychain, or modify code before it is committed.

## Safe IDE Practices

Use these practices when configuring IDEs:

- Install extensions only from trusted publishers.
- Review permissions and requested capabilities.
- Remove unused extensions.
- Keep IDEs and plugins updated.
- Avoid extensions with unclear ownership or poor maintenance.
- Be cautious with extensions that upload code to remote services.
- Disable workspace trust bypasses unless necessary.
- Use organization-managed extension allowlists when available.

### Safe Plugin Practices

- Install plugins only from official marketplaces (VS Code Marketplace, JetBrains Marketplace)
- Check download counts, ratings, and publisher identity before installing
- Review the permissions requested by a plugin — does a spellchecker need network access?
- Periodically audit installed plugins and remove those no longer in use
- Prefer plugins from your organization's approved list if one exists
- Check if the plugin's source code is publicly available and recently maintained

### Risks Associated with IDE Plugins

- **Typosquatting:** A plugin named `prettier-js` may be legitimate; `prettiier-js` may not be.
- **Dependency confusion:** Plugins that pull in packages from public registries may fetch malicious versions if internal package names are not properly scoped.
- **Excessive permissions:** Some plugins request access to system clipboard, network, or file system far beyond what their stated purpose requires.
- **Abandoned plugins:** A plugin that is no longer maintained may contain unpatched vulnerabilities or have its publisher account compromised.

## Language Servers and Build Tools

Language servers, linters, formatters, and build tools can execute code or parse untrusted files. A malicious repository may contain configuration that causes local commands to run.

Examples of risky files or behaviors:

- Build scripts that run automatically.
- Package installation hooks.
- Editor task definitions.
- Workspace-specific settings that change execution behavior.
- Test runners that execute arbitrary code.

Before opening or building an unfamiliar repository:

- Inspect configuration files.
- Avoid running install scripts blindly.
- Use containers or isolated environments for unknown projects.
- Check dependency sources.
- Avoid using production credentials in the same shell session.

## Shell and Terminal Tools

Shell customizations can improve productivity but may introduce risk.

Be cautious with:

- Installing shell scripts directly from the internet using `curl | sh`.
- Unknown dotfiles.
- Terminal themes or plugins that execute code.
- Aliases that override security-sensitive commands.
- Scripts requiring broad permissions.

Safer approach:

1. Download the script.
2. Read it.
3. Verify the source.
4. Run it with the minimum required permissions.
5. Avoid running as root unless necessary.

## Git Hooks

Git hooks can execute local scripts during commits, merges, checkouts, or pushes. They are useful for checks but can also be abused.

Best practices:

- Understand what hooks are installed.
- Do not blindly copy hooks from unknown repositories.
- Use centrally managed hooks for security checks where possible.
- Avoid hooks that collect or transmit unnecessary data.

## Package Managers

Package managers are frequent supply-chain targets. Examples include npm, pip, Maven, Gradle, Go modules, RubyGems, Cargo, NuGet, and system package managers.

Risks include:

- Typosquatting packages.
- Dependency confusion.
- Malicious post-install scripts.
- Compromised maintainers.
- Abandoned packages taken over by attackers.
- Overly broad dependency trees.

Safer habits:

- Use lockfiles.
- Pin versions where appropriate.
- Prefer internal package mirrors or registries.
- Review new dependencies before adding them.
- Avoid packages with suspicious names or low reputation.
- Keep dependencies updated through controlled processes.
- Use dependency scanning tools.

### Local Tools and Package Managers

Development environments often include many tools installed via package managers (Homebrew, apt, Chocolatey, pip, npm, cargo). Each installation is a potential vector.

**Risks:**
- Malicious packages masquerading as popular libraries (typosquatting)
- Packages with legitimate names but malicious `postinstall` scripts
- Globally installed tools that persist across projects and accumulate over time

**Mitigations:**
- Pin versions of tools and dependencies where possible
- Use lock files (package-lock.json, Pipfile.lock, Cargo.lock) and verify them in version control
- Audit global package installations regularly
- Use tools like `npm audit`, `pip-audit`, or `cargo audit` to check for known vulnerabilities in dependencies

# 16.5 Use of Online Services and “Free” Tools

Online services can be helpful, but developers must understand what data they are sharing and where it goes. Free tools are not automatically unsafe, but they often come with tradeoffs involving data retention, analytics, training, advertising, or limited contractual protection.

### The "Free" Tool Problem

When a service is offered for free, ask: what is the business model? In many cases, the product is the data you provide. For developer tools, this can mean your code, your environment variables, your API responses, or your internal hostnames are being collected, stored, and potentially shared.

## Common Online Tools Developers Use

Developers may use online services for:

- JSON, YAML, XML, or SQL formatting.
- Regex testing.
- JWT decoding.
- Certificate inspection.
- Code conversion.
- Diagram generation.
- Log analysis.
- AI coding assistance.
- Translation.
- Screenshot annotation.
- File conversion.
- Debugging or stack trace analysis.

Sites like Pastebin, GitHub Gist (public), and Hastebin are convenient but permanent and public by default. Sensitive information committed to these services may be indexed by search engines or scraped by automated tools.

**Free VPNs**
Free VPN services frequently monetize by logging and selling traffic data. Using a free VPN to tunnel work traffic is potentially worse than not using one at all.

## What Not to Paste into Online Tools

Do not paste the following into unapproved online services:

- API keys, tokens, passwords, private keys, certificates, or secrets.
- JWTs, session cookies, OAuth codes, or refresh tokens.
- Proprietary source code.
- Customer data or production records.
- Internal logs containing identifiers, credentials, IPs, URLs, or payloads.
- Security findings or vulnerability details.
- Database dumps.
- Incident reports.
- Configuration files containing endpoints, account IDs, or infrastructure details.

## Online Tool Risk Factors

Before using an online tool, consider:

- Does the service store submitted data?
- Is data used for model training, analytics, or advertising?
- Is there an enterprise agreement or data processing agreement?
- Is traffic encrypted?
- Who operates the service?
- Is the tool approved by the organization?
- Can the same task be done locally using a trusted tool?

Pasting code into an online tool (JSON formatter, regex tester, SQL prettifier) may transmit that code to a third party. If the code contains credentials, customer data, or proprietary logic, this is a data disclosure risk.

## Safer Alternatives

Instead of free online tools, prefer:

- Local formatters and validators.
- Organization-approved SaaS tools.
- Internal developer portals.
- Enterprise AI tools with approved data handling controls.
- Offline scripts for common transformations.
- Sanitized examples that remove secrets and sensitive data.

## AI Coding Assistants

AI tools can improve productivity, but developers must use them carefully.

Recommended practices:

- Follow organizational AI usage policy.
- Do not paste secrets or sensitive data.
- Avoid sharing proprietary code with unapproved services.
- Review generated code for security, licensing, correctness, and maintainability.
- Treat generated dependencies and commands as untrusted until reviewed.
- Avoid asking AI tools to process production logs unless approved and sanitized.

Generated code is not automatically secure. Developers remain responsible for review, testing, and compliance.

Tools such as GitHub Copilot, ChatGPT, and others may transmit code snippets (or entire files) to remote servers for processing. This creates risks around:
- Inadvertent disclosure of proprietary algorithms or business logic
- Submission of credentials, tokens, or PII embedded in code
- Regulatory non-compliance if data processed is subject to GDPR, HIPAA, or similar

Check your organization's policy before using AI coding assistants on company code. Many enterprises have negotiated enterprise agreements with vendors that include data residency and privacy commitments.

### A Simple Decision Framework

Before using an online tool with company data, ask:

1. Does my organization's policy permit this?
2. Does the tool's privacy policy allow them to retain, analyze, or share my input?
3. Would I be comfortable if this data appeared in a data breach disclosure?
4. Is there an approved internal or enterprise alternative?

If the answer to any of these makes you uncomfortable, do not use the tool.

# 16.6 Secrets Management and Sensitive Data Handling

Secrets are among the most common and damaging exposures in development environments.

## What Counts as a Secret?

Examples include:

- Passwords.
- API keys.
- SSH private keys.
- TLS private keys.
- Database credentials.
- Cloud access keys.
- OAuth client secrets.
- Personal access tokens.
- Webhook secrets.
- Signing keys.
- Encryption keys.
- Session cookies.
- Refresh tokens.
- kubeconfig files.
- Service account credentials.

## Secrets Handling Principles

### Never Hardcode Secrets

Do not place secrets in source code, scripts, configuration templates, test files, documentation, or comments.

Hardcoding secrets directly into source code is one of the most frequently observed and consequential developer security mistakes. Once a secret is committed to version control, it is effectively compromised — even if the commit is later deleted, it may have been cached, forked, or scanned.

Automated tools (GitHub secret scanning, TruffleHog, GitLeaks) actively scan public and private repositories for patterns matching known credential formats. Attackers run similar tools continuously.

**Never do this:**
```python
# BAD — never hardcode secrets
db_password = "MyP@ssw0rd123"
api_key = "sk-live-abc123xyz"
```

**Do this instead:**
```python
# GOOD — load from environment or secrets manager
import os
db_password = os.environ.get("DB_PASSWORD")
api_key = os.environ.get("API_KEY")
```


### Avoid Long-Lived Credentials

Use short-lived, scoped, and revocable credentials where possible.

### Use Approved Secret Stores

Prefer managed secret systems such as organization-approved vaults, cloud secret managers, or secure CI/CD secret stores.

### Restrict Scope

A development token should not have production administrator permissions. A token for reading one repository should not access all repositories.

### Rotate After Exposure

If a secret is exposed, assume compromise. Remove it from where it appeared, revoke or rotate it, and review logs for misuse.

## Local `.env` Files

`.env` files are convenient but risky. They are often accidentally committed or shared.

Best practices:

- Add `.env` to `.gitignore`.
- Commit `.env.example` with placeholder values only.
- Do not store production secrets in local `.env` files.
- Use separate environment files per context.
- Delete stale files.
- Avoid syncing `.env` files to personal cloud storage.

## Git and Secret Exposure

Secrets committed to Git remain in history even if deleted in a later commit. Treat committed secrets as compromised.

When a secret is committed:

1. Stop using the secret.
2. Revoke or rotate it.
3. Remove it from current code.
4. Follow the organization’s Git history cleanup process if required.
5. Notify security or platform teams.
6. Check logs for suspicious use.

## Sensitive Data in Logs

Logs often contain sensitive information by accident.

Avoid logging:

- Passwords or tokens.
- Authorization headers.
- Cookies.
- Full request or response bodies with personal data.
- Payment data.
- Private keys.
- Internal identifiers when not needed.

When debugging, sanitize logs before sharing.

- **Do not use real production data in development or test environments.** Use synthetic or anonymized data.
- **Do not log sensitive values.** Ensure that passwords, tokens, and PII are not written to application logs.
- **Rotate secrets regularly** and immediately upon suspected compromise.
- **Scope secrets to the minimum necessary permissions** — an API key used only to read data should not have write or delete permissions.

## Test Data

Use synthetic or anonymized data whenever possible. Production data should not be copied to local environments unless explicitly approved and protected.

Safe test data should:

- Not identify real customers or employees.
- Not contain valid credentials.
- Not include live payment, health, or regulated data.
- Be clearly marked as test data.

### Secrets Management Tools

| Tool | Use Case |
|---|---|
| **HashiCorp Vault** | Enterprise secrets management, dynamic credentials |
| **AWS Secrets Manager** | Managed secrets for AWS workloads |
| **Azure Key Vault** | Managed secrets for Azure workloads |
| **GCP Secret Manager** | Managed secrets for GCP workloads |
| **1Password Secrets Automation** | Developer-friendly secrets injection |
| **Doppler** | Secrets sync across environments |
| **.env files (local only)** | Acceptable for local development only; never commit |

### Pre-commit Hooks for Secret Detection

Install a pre-commit hook that scans staged files for secrets before allowing a commit:

```bash
# Install detect-secrets
pip install detect-secrets

# Initialise baseline
detect-secrets scan > .secrets.baseline

# Install pre-commit hook
pre-commit install
```

Tools such as `git-secrets`, `gitleaks`, and `detect-secrets` can be integrated into your local workflow and CI/CD pipeline.

# 16.7 Network, Remote Access, and Home Office

Remote and hybrid work can be secure when basic controls are followed. The goal is to prevent unauthorized access, network interception, device compromise, and accidental exposure.

## Home Network Security

Recommended practices:

- Use WPA2 or WPA3 Wi-Fi encryption.
- Change default router passwords.
- Keep router firmware updated.
- Use a separate guest network for visitors and IoT devices.
- Avoid exposing local development services to the internet.
- Disable unnecessary router features such as remote administration.

Developers often work across multiple network environments — corporate offices, home offices, co-working spaces, and occasionally coffee shops. Each environment presents different risks.

| Environment | Key Risks |
|---|---|
| Corporate office | Relatively controlled; still vulnerable to insider threats and misconfigured internal services |
| Home office | Router may be unpatched; shared with personal/IoT devices; no corporate monitoring |
| Public Wi-Fi | Unencrypted traffic interception; rogue access points; other users on same segment |
| Mobile hotspot | Generally safer than public Wi-Fi; subject to carrier interception |



## Public Wi-Fi

Public Wi-Fi increases risk. When using it:

- Use approved VPN or secure access tools.
- Avoid accessing sensitive systems if the network seems suspicious.
- Confirm you are connecting to the correct network.
- Do not ignore certificate warnings.
- Avoid local file sharing or open services.

## VPN and Zero Trust Access

Remote access tools should be used according to policy. Developers should not create unofficial tunnels or backdoors to internal systems.

Avoid:

- Sharing VPN credentials.
- Saving credentials in insecure scripts.
- Keeping long-lived sessions open unnecessarily.
- Bypassing approved access controls with personal tools.
- Exposing internal tools through public tunnels without approval.

A VPN (Virtual Private Network) encrypts traffic between your workstation and the corporate network. This is particularly important when:
- Working from home or on public Wi-Fi
- Accessing internal services not exposed to the public internet
- Handling sensitive data that should not traverse untrusted networks

**Best practices:**
- Always connect to the corporate VPN before accessing internal resources
- Do not use split tunnelling that routes sensitive traffic outside the VPN
- Verify VPN connection before beginning work on sensitive tasks
- Report VPN connectivity issues to IT rather than working around them

## SSH Security

SSH is powerful and commonly used by developers.

Best practices:

- Use strong key pairs.
- Protect private keys with passphrases.
- Store keys securely.
- Use separate keys for different contexts when appropriate.
- Remove old keys from authorized systems.
- Do not share private keys.
- Avoid agent forwarding unless necessary.
- Be cautious with unknown hosts and host key warnings.

SSH key pairs are a critical credential for developers accessing servers, version control, and cloud infrastructure.

- Generate a new key pair for each purpose (work vs personal vs production access) rather than reusing one key everywhere
- Protect private keys with a strong passphrase
- Store private keys only on your local workstation — never on shared servers or in source control
- Use `ssh-agent` with a timeout rather than leaving keys loaded indefinitely
- Rotate keys when staff leave, keys are compromised, or on a scheduled basis

```bash
# Generate a strong Ed25519 key pair
ssh-keygen -t ed25519 -C "your.email@company.com" -f ~/.ssh/id_ed25519_work

# Add to agent with 8-hour timeout
ssh-add -t 28800 ~/.ssh/id_ed25519_work
```


## Local Services

Developers frequently run local servers, databases, dashboards, and debugging tools.

Secure habits:

- Bind local services to `localhost` unless remote access is required.
- Do not expose development ports publicly.
- Use authentication for local admin dashboards when available.
- Shut down services when finished.
- Avoid running services with production credentials.

## Home Office Physical Security

Basic physical controls matter:

- Lock the screen when away.
- Do not leave devices unattended in public places.
- Store hardware tokens securely.
- Avoid visible sensitive information during video calls or screen sharing.
- Use privacy screens where appropriate.
- Keep family members, guests, and roommates from using work devices.

Home network is an extension of work environment when working from it. Apply the same rigour:

- Change the default admin credentials on your home router
- Ensure your router firmware is up to date
- Use WPA3 (or WPA2 minimum) with a strong Wi-Fi password
- Separate work devices from personal and IoT devices using a guest network or VLAN
- Do not allow other household members to use your work devices

# 16.8 Containerized / Virtual Development Environments

Containers and virtual machines can improve isolation, repeatability, and dependency management. However, they are not automatically secure.

## Benefits

Containerized or virtual development environments can:

- Reduce host pollution from tools and dependencies.
- Make environments reproducible.
- Limit the impact of untrusted code.
- Separate project dependencies.
- Support consistent onboarding.
- Reduce conflicts between projects.

Containers and virtual machines provide isolation between your development environment and your host operating system. This has significant security benefits:

- **Blast radius reduction:** If a container is compromised, the impact is (in theory) limited to that container
- **Reproducibility:** Defined, version-controlled environments reduce the risk of "works on my machine" configuration drift
- **Ephemeral environments:** Containers can be destroyed and recreated, eliminating persistent malware
- **Dependency isolation:** Each project can have its own pinned dependencies without polluting the host


## Risks

Potential risks include:

- Running containers as privileged.
- Mounting sensitive host directories.
- Exposing Docker socket access.
- Using untrusted base images.
- Running containers as root.
- Pulling images from unknown registries.
- Storing secrets in images or Dockerfiles.
- Publishing images that contain secrets or proprietary code.

## Docker Socket Risk

Mounting the Docker socket into a container can effectively give the container control over the host’s container runtime. This may allow privilege escalation or host compromise.

Avoid mounting:

```bash
/var/run/docker.sock
```

unless there is a clear, approved reason.

## Privileged Containers

Avoid running containers with:

```bash
--privileged
```

Privileged containers weaken isolation and can expose host resources.

## Host Mounts

Be careful with volume mounts. Mounting the entire home directory into a container may expose SSH keys, cloud credentials, Git configuration, browser files, and source code.

Prefer narrow mounts:

```bash
docker run -v "$PWD:/workspace" image-name
```

Avoid broad mounts:

```bash
docker run -v "$HOME:/host-home" image-name
```

## Image Hygiene

Use trusted images and keep them updated.

Best practices:

- Prefer official or organization-approved base images.
- Pin image versions or digests where appropriate.
- Scan images for vulnerabilities.
- Avoid installing unnecessary packages.
- Do not bake secrets into images.
- Use multi-stage builds to reduce final image size and contents.
- Run as a non-root user inside the container where possible.

## Development Containers

Dev containers, cloud workspaces, and remote IDEs can be helpful, especially when centrally managed.

Security considerations:

- Confirm where code and data are stored.
- Understand whether the environment has internet access.
- Control access to workspace snapshots and logs.
- Use approved base images and templates.
- Rotate secrets used in ephemeral environments.
- Avoid leaving unused environments running indefinitely.

## Virtual Machines

VMs provide stronger isolation than ordinary containers in many scenarios, but they still need patching and configuration.

Recommended practices:

- Keep VM images updated.
- Avoid copying secrets between host and guest unnecessarily.
- Use snapshots carefully; snapshots may contain secrets.
- Delete unused VMs.
- Restrict shared clipboard and shared folders for untrusted environments.

Running containers introduces its own attack surface if misconfigured.

**Never run containers as root unless absolutely necessary:**
```dockerfile
# BAD
FROM node:18
# runs as root by default

# GOOD
FROM node:18
RUN useradd -m appuser
USER appuser
```

**Do not mount sensitive host directories into containers:**
```bash
# DANGEROUS — mounts entire home directory
docker run -v $HOME:/home myimage

# BETTER — mount only what is needed
docker run -v $(pwd)/data:/app/data:ro myimage
```

**Use read-only mounts where possible** (note the `:ro` flag above).

**Scan images for vulnerabilities before use:**
```bash
# Using Trivy (open source)
trivy image myimage:latest

# Using Docker Scout
docker scout cves myimage:latest
```

**Keep base images up to date and prefer minimal base images** (e.g. `alpine`, `distroless`) to reduce attack surface.

### Docker Socket Risk

Mounting the Docker socket (`/var/run/docker.sock`) into a container gives that container effective root access to the host. This is a common misconfiguration in CI/CD pipelines.

```bash
# DANGEROUS — do not do this
docker run -v /var/run/docker.sock:/var/run/docker.sock myimage
```

### Dev Containers and Codespaces

Tools like VS Code Dev Containers and GitHub Codespaces provide containerized development environments that can be version-controlled alongside your project code. Security benefits include:

- The development environment is defined in code and reviewable
- Developers cannot accidentally install unapproved tools on the host
- Environments can be centrally governed and updated

# 16.9 Developer Awareness and Secure Habits

Secure habits reduce risk more reliably than occasional security reminders.

## Daily Habits

Developers should regularly:

- Lock their screen when away.
- Review commands before running them.
- Avoid copying commands from unknown sources without understanding them.
- Keep tools and dependencies updated.
- Use MFA for important services.
- Check Git diffs before committing.
- Avoid committing generated files that may contain secrets.
- Sanitize logs and screenshots before sharing.
- Remove old branches, credentials, and temporary files.

## Before Running Code from a New Repository

Ask:

- Who owns this repository?
- Do I trust the source?
- What install scripts will run?
- What files will it read or modify?
- Does it require credentials?
- Can I run it in an isolated environment?

## Before Installing a Dependency

Ask:

- Is the package name correct?
- Is it maintained?
- Is the publisher trustworthy?
- Does it have suspicious install scripts?
- Are there safer internal alternatives?
- Does it introduce many transitive dependencies?

## Before Sharing Debug Information

Ask:

- Does this contain tokens, cookies, or authorization headers?
- Does it include customer or employee data?
- Does it reveal internal endpoints or architecture?
- Is this going to an approved channel or tool?
- Can I redact or minimize it?

## Before Using AI or Online Tools

Ask:

- Is this tool approved for this type of data?
- Am I sharing proprietary code?
- Am I sharing secrets or logs?
- Can I sanitize the input?
- Can I do this locally instead?

## Security Mindset

Security is not about saying “no” to developer productivity. It is about making safe paths easy and risky paths visible. Good developer security balances speed, autonomy, and protection.

A useful mindset is:

- Assume credentials can leak.
- Assume dependencies can be compromised.
- Assume local tools can access more than expected.
- Assume logs contain sensitive data until proven otherwise.
- Assume production access requires extra care.

### Security is a Habit, Not an Audit

Security controls are most effective when they become reflexive habits rather than conscious checklists. The goal of awareness is to shift your instinctive response so that you notice something suspicious before it becomes an incident.

### Recognizing Social Engineering

Developers are increasingly targeted by sophisticated social engineering attacks, including:

- **Spear phishing:** Emails crafted to appear from a colleague, your GitHub account team, or a cloud provider, requesting credential confirmation or inviting you to click a link
- **Pretexting:** A request that seems to come from a legitimate context ("I'm from IT, I need your login to fix an issue")
- **Malicious pull requests:** Contributions to open source projects that introduce subtle backdoors, often from accounts with established history
- **Dependency confusion / typosquatting:** Exploiting how developers search for and install packages

A useful rule: **legitimate services never ask for your password**. If something requests credentials unexpectedly, treat it as suspicious and verify through a separate channel.

### Code Review as a Security Control

Code review is one of the most effective security controls available to a development team. When reviewing code:

- Look for hardcoded secrets, credentials, or sensitive values
- Check that input validation is present for all user-supplied data
- Verify that error messages do not expose stack traces or internal details
- Ensure that new dependencies are intentional and from expected sources
- Check for changes to authentication, authorization, or cryptographic logic — these deserve extra scrutiny

### Commit Hygiene

- Sign your commits with a GPG or SSH key to provide authenticity (`git commit -S`)
- Write meaningful commit messages — they create an audit trail
- Avoid committing large batches of unrelated changes — small, focused commits are easier to review and audit
- Use `.gitignore` to prevent accidental commits of `.env` files, private keys, and build artifacts

### Staying Current

The threat landscape changes constantly. Dedicate time to staying informed:

- Follow CVE feeds for languages and frameworks you use (e.g. Python advisory database, npm security advisories)
- Subscribe to your cloud provider's security bulletins
- Follow CISA's Known Exploited Vulnerabilities (KEV) catalog
- Participate in your organization's security training and drills

# 16.10 Practical Checklists and Self-Assessment

## Workstation Baseline Checklist

Use this checklist to confirm that your workstation meets basic security expectations.

- [ ] Full-disk encryption is enabled.
- [ ] Operating system updates are installed regularly.
- [ ] Endpoint protection or EDR is running as required.
- [ ] Screen lock is enabled with a short timeout.
- [ ] Local account uses a strong password or approved authentication method.
- [ ] MFA is enabled for code hosting, cloud, SSO, and package registry accounts.
- [ ] Administrator or root access is not used for routine work.
- [ ] Unused local accounts have been removed.
- [ ] Device is enrolled in required management tools.
- [ ] Lost-device and remote-wipe procedures are understood.

## IDE and Plugin Checklist

- [ ] IDE is up to date.
- [ ] Extensions are from trusted publishers.
- [ ] Unused extensions are removed.
- [ ] Extensions that upload code or telemetry are approved.
- [ ] Workspace trust settings are understood.
- [ ] Project-specific scripts and tasks are reviewed before execution.
- [ ] AI coding tools are used according to policy.

## Secrets Checklist

- [ ] No secrets are hardcoded in source code.
- [ ] `.env` files are ignored by Git.
- [ ] `.env.example` contains placeholders only.
- [ ] Local secrets are scoped and non-production where possible.
- [ ] Secrets are stored in approved secret managers.
- [ ] Long-lived tokens are avoided where possible.
- [ ] Exposed secrets are reported and rotated immediately.
- [ ] Git commits are checked for accidental secret inclusion.

## Dependency Checklist

- [ ] New dependencies are reviewed before adoption.
- [ ] Package names are verified to avoid typosquatting.
- [ ] Lockfiles are used where appropriate.
- [ ] Dependency scanning is enabled.
- [ ] Deprecated or abandoned packages are replaced.
- [ ] Package manager tokens have minimum required scope.
- [ ] Internal package registries are used where required.

## Online Tool Checklist

- [ ] No secrets are pasted into unapproved online tools.
- [ ] No proprietary code is submitted to unapproved services.
- [ ] Logs are sanitized before sharing.
- [ ] Customer or employee data is not uploaded without approval.
- [ ] Approved enterprise tools are used when available.
- [ ] Free tools are reviewed for data retention and privacy concerns.

## Remote Work Checklist

- [ ] Home Wi-Fi uses WPA2 or WPA3.
- [ ] Router default password has been changed.
- [ ] Router firmware is kept updated.
- [ ] Work device is not shared with family or guests.
- [ ] VPN or approved secure access is used when required.
- [ ] Public Wi-Fi is avoided for sensitive work where possible.
- [ ] Local development services are not exposed publicly.
- [ ] Screen sharing is checked for sensitive information before meetings.

## Container and VM Checklist

- [ ] Containers are not run with `--privileged` unless approved.
- [ ] Docker socket is not mounted into containers unless approved.
- [ ] Host volume mounts are narrow and intentional.
- [ ] Base images are trusted and updated.
- [ ] Images are scanned for vulnerabilities.
- [ ] Secrets are not baked into images.
- [ ] Containers run as non-root where possible.
- [ ] Unused containers, images, volumes, and VMs are removed.
- [ ] VM snapshots containing secrets are protected or deleted.
