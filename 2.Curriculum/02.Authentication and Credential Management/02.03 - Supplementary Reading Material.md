# Supplementary Reading Material - Authentication and Credential Management

Authentication and Credential Management — Exam Reading Material for Developers

## 2.1 Core Authentication Concepts

### What authentication is, and is not

Authentication is the process of verifying the identity of a user, service, device, or application. It answers one question:

**Who are you?**

Authentication does not answer:

- What the authenticated entity is allowed to do
- What data it can access
- Whether a specific action should be permitted

Those are authorization questions.

This distinction matters because a secure system must do both:

- Authenticate the subject
- Authorize the requested action

A common design mistake is assuming that once a user is logged in, all actions are safe. A valid login only proves identity. It does not grant unlimited access.

### Available authentication methods

Authentication methods are usually grouped by factor type.

#### Something you know

- Password
- PIN
- Passphrase
- Recovery code

These are widely used but can be guessed, reused, phished, or leaked.

#### Something you have

- Authenticator app
- Hardware token
- Smart card
- Security key
- Trusted device
- Client certificate

These are generally stronger than knowledge factors alone.

#### Something you are

- Fingerprint
- Face recognition
- Iris scan

Biometric factors are convenient, but if compromised they cannot be changed like passwords.

### Common authentication mechanisms in systems

- Username and password
- Password plus MFA
- Passkeys
- Magic links
- API keys
- OAuth or OIDC login through an identity provider
- Mutual TLS for service-to-service communication
- Signed tokens or assertions

Developers should choose methods based on risk, usability, and the type of entity being authenticated.

### Entities requiring authentication

Authentication is not limited to human users.

#### Human users

- Customers
- Employees
- Administrators
- Support staff

#### Non-human identities

- Backend services
- Microservices
- Applications
- Scheduled jobs
- Deployment pipelines
- Containers
- Service accounts

#### Devices and systems

- Mobile devices
- IoT devices
- Administrative consoles
- Network appliances

Each entity type may need a different authentication method. For example:

- Humans often use passwords, MFA, or passkeys
- Services may use certificates or signed tokens
- Third-party integrations may use OAuth client credentials or API keys

A secure design starts with identifying all actors that must prove identity.

## 2.2 Credential Lifecycle Management

Credentials must be managed throughout their lifecycle—from creation to retirement. Security depends not only on how credentials are used, but also on how they are issued, stored, rotated, recovered, and revoked.

### Credential provisioning

Provisioning is the creation and secure issuance of a credential.

Examples:

- Creating a password during user registration
- Issuing an API key to a client
- Generating a certificate for a service
- Enrolling a hardware token
- Registering a passkey

Good provisioning practices:

- Verify identity before issuing the credential
- Deliver secrets over secure channels
- Show sensitive secrets only when necessary
- Bind the credential to a known subject and purpose
- Record metadata such as owner, issue date, and expiry
- Store passwords only as strong password hashes

For passwords, developers should use modern password hashing algorithms such as:

- Argon2id
- scrypt
- bcrypt

Plain hashing with general-purpose hash functions is not sufficient for password storage. Using salt provides an additional layer of security.

### Credential rotation

Rotation is the replacement of a credential with a new one.

Rotation reduces risk when:

- A credential ages
- A credential may be exposed
- Policy requires periodic replacement
- Cryptographic material is nearing expiry

Credentials that are often rotated include:

- API keys
- Client secrets
- Certificates
- Signing keys
- Database passwords
- Service account secrets

Secure rotation practices:

- Support overlap between old and new credentials
- Automate rotation where possible
- Avoid downtime during replacement
- Monitor for stale credentials
- Log who rotated the credential and why

Emergency rotation should also be supported after suspected compromise.

### Credential revocation

Revocation is the invalidation of a credential before its normal expiry.

Examples:

- Disabling a user account
- Revoking a refresh token
- Invalidating a session
- Disabling an API key
- Revoking a certificate

Revocation is required when:

- A credential is leaked
- A device is lost
- An account is compromised
- An employee leaves
- Policy requires immediate access removal

A secure system should not depend only on expiry. Some credentials must be stoppable immediately.

### Secure credential recovery

Recovery allows legitimate users to regain access when credentials are lost or forgotten.

This is often one of the weakest parts of an authentication system because attackers may target recovery instead of the primary login flow.

Secure recovery principles:

- Treat recovery as a high-risk process
- Verify identity carefully
- Use short-lived, single-use recovery tokens
- Send recovery links through trusted channels
- Notify the user when recovery occurs
- Invalidate old recovery artifacts after use
- Require re-authentication before sensitive changes

Avoid:

- Weak security questions
- Predictable reset links
- Indefinite recovery tokens
- Recovery processes that rely only on publicly known information

For privileged accounts, recovery should require stronger controls than for standard users.

### Password best practices

Passwords remain common, so developers must implement them safely.

Best practices:

- Allow long passwords and passphrases
- Support password managers
- Allow paste into password fields
- Block commonly used and breached passwords
- Hash passwords securely
- Apply login rate limiting
- Monitor for credential stuffing attacks

Practices to avoid:

- Storing passwords in plaintext
- Logging passwords
- Forcing arbitrary complexity rules that reduce usability
- Silently truncating passwords
- Forcing frequent password changes without cause
- Using knowledge-based questions as the main recovery defense

A strong password policy emphasizes:

- Length
- Uniqueness
- Resistance to known breached passwords
- Secure storage
- MFA support

## 2.3 Multi-Factor Authentication (MFA)

MFA requires two or more factors from different categories.

Examples:

- Password plus authenticator app code
- Password plus hardware security key
- Passkey plus device possession and local biometric unlock

MFA is valuable because a compromised password alone is no longer enough.

### Common MFA methods

#### Authenticator app codes

These are stronger than password-only login and better than many SMS-based approaches, but users can still be tricked into entering them on phishing sites.

#### SMS codes

These improve security compared with password-only login, but they are weaker than phishing-resistant methods and can be vulnerable to SIM swapping and interception.

#### Push notifications

Convenient, but users may approve requests by mistake. Safer designs include number matching or visible transaction context.

#### Hardware security keys

Strong, phishing-resistant, and well suited for privileged users and administrators.

#### Backup codes

Useful as a fallback, but they must be high entropy, shown securely, and invalidated after use.

### MFA design principles

Use MFA especially for:

- Administrator accounts
- Remote access
- Sensitive data access
- Privileged actions
- Account recovery
- High-risk transactions

Also consider step-up authentication. This means requiring additional verification only when risk or sensitivity increases.

Examples:

- Changing MFA settings
- Changing payout information
- Accessing production systems
- Viewing highly sensitive records

Weak fallback paths can completely undermine MFA. Secure MFA must include secure enrollment, reset, and recovery processes.

## 2.4 Session Management

After authentication succeeds, the application usually creates a session. Session management controls how that authenticated state is maintained and protected.

A secure login is not enough if the session is easy to steal or misuse.

### Session timeout and renewal

Sessions should expire appropriately.

Common session controls:

- Idle timeout: session expires after inactivity
- Absolute timeout: session ends after a maximum lifetime
- Session renewal: session identifier is replaced periodically
- Re-authentication: user must prove identity again for sensitive actions

Long-lived sessions improve convenience but increase exposure if stolen.

Sensitive actions may require re-authentication even when the session is still active.

### Cookie attributes

When using cookies to store session state or session identifiers, cookie attributes are a major security control.

#### Secure

The cookie is sent only over HTTPS. This helps prevent exposure over insecure transport.

#### HttpOnly

The cookie cannot be read through client-side JavaScript. This reduces the impact of some cross-site scripting attacks.

#### SameSite

Controls when cookies are included in cross-site requests.

Values:

- `Strict` — strongest restriction
- `Lax` — commonly used balance
- `None` — allows cross-site sending, must also use `Secure`

`SameSite` helps reduce CSRF risk, but it must be chosen carefully to avoid breaking intended flows.

### Token vs. cookie-based sessions

Developers should understand that these are different mechanisms, each with trade-offs.

#### Cookie-based sessions

A session identifier is stored in a cookie, and session state is often kept server-side.

Advantages:

- Common in traditional web applications
- Simple invalidation when server stores session state
- Works naturally with browsers

Risks:

- CSRF if not defended properly
- Session theft if cookie protections are weak

#### Token-based sessions

The client stores and sends a token, often in an authorization header.

Advantages:

- Useful for APIs and mobile clients
- Can fit distributed systems well
- May reduce server-side session storage needs

Risks:

- Difficult revocation if poorly designed
- Token theft if stored unsafely
- Bearer tokens are usable by anyone who possesses them

Important point:

A JWT is only a token format. It is not automatically secure. Security depends on correct storage, validation, lifetime control, and transmission.

### Logout and re-authentication

Logout should terminate the session in a meaningful way.

Secure logout behavior:

- Invalidate the server-side session if one exists
- Remove session cookies or tokens from the client
- Revoke refresh tokens where applicable
- Support logout from all devices where required

A weak logout design may remove the local token but leave the server-side authenticated state active.

Re-authentication is important before:

- Password changes
- MFA enrollment or reset
- Email address changes
- Privileged administrative actions
- Destructive or financial operations

### Fixation and hijacking defenses

#### Session fixation

An attacker tricks a user into authenticating with a session identifier already known to the attacker.

Defense:

- Issue a new session identifier after login
- Rotate session identifiers after privilege changes

#### Session hijacking

An attacker steals or reuses a valid session.

Defenses:

- HTTPS everywhere
- Secure cookie settings
- Short session lifetime
- Session rotation
- Re-authentication for sensitive actions
- Anomaly detection
- Strong XSS defenses
- Ability to terminate sessions remotely

Session security is a core part of authentication security.

## 2.5 Federation and Single Sign-On (SSO)

Federation allows one system to rely on another trusted identity system for authentication.

SSO allows a user to authenticate once and then access multiple systems without separate logins.

This improves usability and centralizes identity management, but it also concentrates trust.

### Authentication vs. Authorization

In federated environments, this distinction is especially important.

The identity provider may confirm:

- Who the user is
- When the user authenticated
- Which authentication method was used
- Claims about the user

The application must still decide:

- What the user may do
- What data the user may access
- What roles or scopes apply

Do not automatically convert identity claims into application privilege without policy checks.

### Current technologies — OIDC and SAML

#### OpenID Connect (OIDC)

OIDC is an identity layer built on OAuth 2.0.

It is widely used for:

- Web applications
- Mobile applications
- Modern identity platforms
- API ecosystems

OIDC commonly uses:

- JSON
- HTTPS
- JWTs
- ID tokens
- Discovery metadata
- JWKs for key distribution

#### SAML

SAML is an older federation standard still common in enterprise systems.

It typically uses:

- XML assertions
- Browser redirects or POSTs
- Digital signatures
- Identity provider and service provider metadata

OIDC is more common in modern developer ecosystems, while SAML remains important in enterprise SSO.

### Signatures and algorithms

Federation depends on trust in signed tokens or assertions.

Developers must validate:

- The issuer
- The audience
- The signature
- The expected algorithm
- Expiry and validity window
- Replay-sensitive identifiers when applicable

Common mistakes:

- Accepting weak or unexpected algorithms
- Trusting unsigned objects
- Skipping issuer or audience checks
- Failing to rotate or validate signing keys properly

Token signature validation is a critical security control. A failure here can allow token forgery or impersonation.

### Mandatory fields on tokens that include security relevance

Exact required fields depend on protocol and token type, but common security-relevant fields include:

#### In OIDC or JWT contexts

- `iss` — issuer
- `sub` — subject
- `aud` — audience
- `exp` — expiration time
- `iat` — issued at
- `nbf` — not before, when used
- `jti` — unique token identifier, when replay detection matters
- `nonce` — important in browser-facing OIDC flows

Applications should reject tokens with missing, invalid, or unexpected critical fields.

#### In SAML contexts

Important items typically include:

- Issuer
- Subject or NameID
- Audience restriction
- Validity conditions
- Assertion identifier
- Authentication statement
- Signature information

Security depends not only on the presence of fields, but on correct validation of their meaning.

### Sensitive information in the payload and security implications

A signed token is not necessarily an encrypted token.

This means payload contents may be readable by:

- Clients
- Browsers
- Debugging tools
- Logs
- Intermediaries that see the token

Do not place unnecessary sensitive data in token payloads.

Examples of risky payload content:

- Passwords
- Secrets
- Recovery data
- Full financial information
- Highly sensitive personal data
- Internal-only authorization logic

Risks include:

- Accidental logging
- Browser exposure
- Leakage to third parties
- Long-term storage in monitoring systems

Best practice:

- Include only the minimum claims necessary for the receiving system to do its job.

## 2.6 Test Authentication

Authentication systems must be tested as security-critical functionality.

Testing should cover not only successful login, but failure cases, abuse cases, recovery, lifecycle events, and session security.

### What to test

#### Positive cases

Verify that valid authentication works correctly:

- Valid username and password
- Valid MFA flow
- Correct token validation
- Expected session creation
- Successful federation login

#### Negative cases

Verify that invalid or malicious input is rejected:

- Wrong password
- Expired token
- Invalid signature
- Wrong audience
- Revoked credential
- Reused recovery token
- Disabled account
- Missing second factor

### Abuse and edge cases

Developers should also test:

- Brute-force resistance
- Rate limiting
- Session fixation resistance
- Replay attempts
- CSRF on login-related flows
- Token tampering
- MFA bypass paths
- Open redirect issues in SSO
- Race conditions in rotation or revocation

### Test the full lifecycle

Authentication is more than login. Test:

- Registration and provisioning
- Credential issuance
- Password reset
- MFA enrollment and recovery
- Credential rotation
- Credential revocation
- Session expiry
- Logout
- Re-authentication
- Compromise response

### Test security properties

It is not enough to check that the feature works. Developers should verify that it works securely.

Examples:

- Passwords are hashed properly
- Cookies include correct security attributes
- Sessions rotate after login
- Expired tokens are rejected
- Revoked tokens no longer work
- Logout actually ends access
- Unauthorized claims are not trusted
- Recovery tokens are single use and short-lived

### Use multiple testing approaches

Authentication testing should be included in:

- Unit tests
- Integration tests
- End-to-end tests
- Negative security tests
- Manual testing of recovery and exception flows
- Penetration testing for higher-risk systems

### Test monitoring and incident readiness

Also verify:

- Security events are logged
- Secrets are not logged
- Suspicious login activity can be detected
- Account lockout or response processes work
- Users receive notifications for important account events

## Key Exam Takeaways

Developers should remember these core points:

- Authentication proves identity; authorization determines access.
- All entities, not only users, may require authentication.
- Credentials must be securely provisioned, rotated, revoked, and recovered.
- Password security depends on storage, policy, and attack resistance.
- MFA improves security, but weak fallback paths can defeat it.
- Session security is as important as login security.
- Cookies must be configured securely.
- Federation requires strict validation of issuers, audiences, signatures, and token claims.
- Token payloads should contain minimal sensitive information.
- Authentication must be tested across normal, failure, and attack scenarios.
