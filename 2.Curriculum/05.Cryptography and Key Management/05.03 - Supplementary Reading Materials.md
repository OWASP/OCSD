# Supplementary Reading Materials - Cryptography and Key Management

Cryptography — Exam Reading Material for Developers

## 5.1 What Is Cryptography

Cryptography is the practice of protecting information and systems using mathematical techniques. It is used to secure data when it is stored, transmitted, processed, or verified.

At a high level, cryptography helps transform readable data into a protected form so that only authorized parties can access or verify it.

Cryptography is not a single tool. It is a collection of techniques and mechanisms used to achieve important security goals in modern systems, applications, APIs, devices, and communications.

Developers encounter cryptography in many places, including:

- HTTPS and TLS
- Password storage
- Token signing
- File encryption
- Secure messaging
- API authentication
- Certificate validation
- Digital signatures
- Key exchange protocols

Cryptography must be applied correctly to provide security. Using strong algorithms incorrectly can still result in insecure systems.

### Main security properties addressed by cryptography

Cryptography is commonly used to support the following security properties.

#### Confidentiality

Confidentiality means protecting data from unauthorized disclosure.

Examples:

- Encrypting database fields
- Encrypting traffic between a browser and a server
- Encrypting backups
- Encrypting secrets stored in configuration systems

Confidentiality helps ensure that only authorized parties can read protected information.

Confidentiality ensures that data is readable only by parties who are authorized to read it. Encryption provides confidentiality by transforming plaintext into ciphertext that is unintelligible without the correct key. If an attacker intercepts an encrypted communication or steals an encrypted database, the data remains protected as long as the key is not compromised.

#### Integrity

Integrity means protecting data from unauthorized or undetected modification.

Examples:

- Detecting whether a message was altered in transit
- Verifying that a file has not been tampered with
- Confirming that a signed token has not been changed

Integrity does not necessarily hide data, but it allows systems to detect changes.

Integrity ensures that data has not been altered — accidentally or maliciously — since it was created or last authorized. Cryptographic hash functions and message authentication codes (MACs) provide integrity guarantees. If even a single bit of the data changes, the verification will fail.

#### Authenticity

Authenticity means being able to verify the source or origin of data, a message, a system, or an identity claim.

Examples:

- Verifying a signed software update
- Verifying the server identity in TLS
- Confirming that a message came from a trusted sender

Authenticity helps answer:

**Did this really come from the expected source?**

Authenticity ensures that a claimed identity is genuine — that a message actually came from who it claims to have come from, or that a party in a communication is who they say they are. Digital signatures and certificates provide authentication. Without this property, an attacker can impersonate a trusted party, even over an encrypted channel.

#### Non-repudiation

Non-repudiation means providing evidence that a party performed an action or approved data in a way that is difficult to deny later.

This is commonly associated with digital signatures in systems where proof of origin or approval matters.

Examples:

- Signed business approvals
- Digitally signed documents
- Signed software releases

In practice, non-repudiation depends not only on cryptography, but also on identity proofing, key control, logging, and legal or organizational context.

Non-repudiation ensures that a party cannot later deny having performed an action or sent a message. Digital signatures provide non-repudiation: because only the holder of a private key can produce a valid signature, a signed document constitutes cryptographic evidence of the signer's involvement.

#### Key point

Cryptography can support confidentiality, integrity, authenticity, and sometimes non-repudiation, but it does not solve every security problem. It does not replace proper authentication, authorization, secure coding, access control, or safe system design.

Note on what cryptography does not provide: Cryptography protects data at rest and in transit. It does not protect against a compromised application, a stolen key, or a user who has been deceived into surrendering their credentials. Cryptographic tools must be embedded in a broader security architecture.

## 5.2 Uses of Cryptography

Cryptography appears in many forms in software systems. Developers should understand the purpose of the most common building blocks and when they are used.

### Keys

A key is a value used by a cryptographic algorithm to perform an operation such as encryption, decryption, signing, or verification.

The security of many cryptographic systems depends on keeping keys secret, managing them properly, and using them only for their intended purpose.

Common key categories include:

- Symmetric keys
- Asymmetric private keys
- Asymmetric public keys
- Session keys
- Signing keys
- Encryption keys
- Key-encryption keys

#### Symmetric keys

Symmetric cryptography uses the same key for both encryption and decryption.

Advantages:

- Efficient
- Fast
- Well suited for large amounts of data

Risks:

- The secret key must be shared securely
- Anyone with the key can decrypt the data

Examples of common uses:

- Disk encryption
- Database encryption
- Encrypted backups
- TLS session encryption

#### Asymmetric keys

Asymmetric cryptography uses a key pair:

- A public key
- A private key

The public key is shared. The private key must be protected.

Typical uses:

- Key exchange
- Digital signatures
- Certificate-based identity
- Secure communication bootstrapping

### Hashes

A hash function takes input data and produces a fixed-length output, often called a hash value or digest.

A secure cryptographic hash should have properties such as:

- It should be hard to reverse
- It should be hard to find two different inputs with the same output
- A small change in input should produce a very different output

Hashes are used for:

- Integrity checking
- Password hashing systems
- Digital signatures
- File verification
- Certificate structures
- Message authentication constructions

Examples of secure modern hash families include SHA-2 and SHA-3.

Developers should not confuse general-purpose hashing with password hashing. Passwords require specialized password hashing functions such as:

- Argon2id
- scrypt
- bcrypt

A cryptographic hash function takes an input of arbitrary length and produces a fixed-length output called a hash, digest, or checksum. The same input always produces the same output. Critically, the function is designed to be one-way: given the hash, it is computationally infeasible to reconstruct the original input.
A good cryptographic hash function has three properties:

Pre-image resistance: Given a hash value, it is computationally infeasible to find any input that produces it.
Second pre-image resistance: Given an input and its hash, it is computationally infeasible to find a different input with the same hash.
Collision resistance: It is computationally infeasible to find any two different inputs that produce the same hash.

Current secure hash algorithms: SHA-256, SHA-384, SHA-512 (from the SHA-2 family), and SHA-3.
Deprecated and insecure: MD5 and SHA-1 must not be used for any security-sensitive purpose. Both have known collision vulnerabilities.

### Message digest

A message digest is the output of a hash function.

It is used as a compact representation of the original data.

Examples of use:

- Comparing file integrity
- Signing a digest instead of signing a large file directly
- Detecting whether content changed during transmission

A digest alone does not prove who created the data. It only helps detect changes unless combined with other controls like digital signatures or message authentication codes.

A MAC is a short piece of data attached to a message that proves two things simultaneously: the message came from a party holding the secret key (authentication), and it has not been modified in transit (integrity). Both the sender and receiver share the same secret key.

An HMAC (Hash-based MAC) constructs a MAC using a hash function (e.g., HMAC-SHA256). It is the most widely used MAC construction and is well-understood and standardized.\
Important distinction: Encryption provides confidentiality but does not, on its own, guarantee integrity. An attacker can modify ciphertext in ways that produce valid-looking plaintext without knowing the key. Always combine encryption with a MAC or use an authenticated encryption mode (see AEAD below

Authenticated Encryption (AEAD)
Modern best practice is to use an Authenticated Encryption with Associated Data (AEAD) cipher mode, which provides confidentiality and integrity together in a single operation. AES-GCM is the standard choice. It encrypts the data and produces an authentication tag; if the ciphertext or associated data has been tampered with, decryption fails.\
Avoid older cipher modes like ECB and CBC without a MAC. They provide confidentiality only and are vulnerable to tampering.

### X.509 certificates

An X.509 certificate is a structured digital document that binds a public key to an identity, such as:

- A domain name
- A service
- An organization
- A person
- A device

Certificates are widely used in TLS and PKI systems.

A certificate typically includes:

- Subject information
- Public key
- Issuer
- Validity period
- Signature by a certificate authority
- Extensions such as key usage or subject alternative names

Certificates help systems verify identity and establish trust chains.

Examples:

- HTTPS server certificates
- Mutual TLS client certificates
- Code signing certificates
- Device identity certificates

An X.509 certificate is a digitally signed document that binds a public key to an identity (a domain name, an organization, a person). It answers the question: "How do I know that the public key I received actually belongs to who I think it does?"\
Certificates are issued by Certificate Authorities (CAs) — trusted third parties that have verified the identity of the certificate holder and signed the certificate with their own private key. Your browser and operating system ship with a set of trusted root CA certificates. When you connect to https://example.com, your browser verifies that the server's certificate was signed by a trusted CA and that it is valid for that domain.\
A certificate contains: the subject's identity, the subject's public key, the issuing CA's identity, validity dates, and the CA's digital signature over all of the above.\
Certificate chain: Most certificates are not signed directly by a root CA but by an intermediate CA, which is itself signed by the root. Clients verify the full chain up to a trusted root.\
Certificate revocation: Certificates can be revoked before they expire (e.g., if the private key is compromised). Revocation is communicated via Certificate Revocation Lists (CRLs) or the Online Certificate Status Protocol (OCSP).

### Digital signatures

A digital signature is used to prove that data was signed by the holder of a private key and that the data has not been modified since signing.

Digital signatures support:

- Integrity
- Authenticity
- Evidence of approval or origin

Examples of use:

- Signed JWTs
- Signed software packages
- Signed documents
- Signed API requests
- Code signing
- Firmware signing

A valid signature depends on:

- Correct algorithm selection
- Proper private key protection
- Correct validation of certificate or public key trust
- Correct verification of the signed content

A digital signature provides authentication, integrity, and non-repudiation using asymmetric cryptography. The process works as follows:

- The signer computes a hash of the data.
- The signer encrypts the hash with their private key. The result is the signature.
- The verifier decrypts the signature with the signer's public key, recovering the hash.
- The verifier independently computes the hash of the data and compares the two. If they match, the signature is valid.

This guarantees that the data was signed by the holder of the private key and has not been modified since it was signed. Common algorithms: RSA with PSS padding, ECDSA.\
Digital signatures are not the same as HMACs. HMACs require a shared secret and provide authentication only to parties who know the secret. Digital signatures use asymmetric keys and allow anyone with the public key to verify, making non-repudiation possible.

### Key Derivation Functions (KDFs)
A KDF derives one or more cryptographic keys from a source of keying material — a passphrase, a shared secret from a key exchange, or another key. KDFs add salt (a random value) and apply computational work to make brute-force derivation expensive. PBKDF2, bcrypt, Argon2, and HKDF are common KDFs, used for different purposes (password storage vs. key derivation from existing secrets).

### Message authentication codes (MACs)

A MAC is used to verify both integrity and authenticity when two parties share a secret key.

Unlike a plain hash, a MAC depends on a secret.

Examples:

- HMAC on API requests
- Signed webhook payloads using shared secrets
- Internal message verification

### Encryption

Encryption converts plaintext into ciphertext using a cryptographic algorithm and a key.

Common uses:

- Protecting data at rest
- Protecting data in transit
- Encrypting backups
- Protecting secrets

Encryption must be paired with correct key management. Encrypted data is not secure if the decryption key is exposed.

### Randomness and nonce values

Many cryptographic systems depend on secure random values for:

- Key generation
- Salts
- Initialization vectors
- Nonces
- Challenge values
- Token generation

Weak randomness can break otherwise strong cryptographic systems.

Developers should use cryptographically secure random number generators, not general-purpose pseudo-random functions intended for simulation or convenience.

## 5.3 Common Algorithm Pitfalls

Cryptographic failure often comes from poor choices or incorrect implementation rather than the basic idea of using cryptography.

### Cipher suites

A cipher suite is a defined combination of cryptographic algorithms used together in protocols such as TLS.

A cipher suite may include choices for:

- Key exchange
- Authentication
- Encryption
- Integrity protection

Developers and operators should avoid weak or outdated cipher suites.

Common problems include:

- Allowing obsolete protocol versions
- Supporting outdated key exchange mechanisms
- Supporting weak ciphers for compatibility
- Enabling insecure fallback behavior

Modern systems should use secure protocol and configuration defaults rather than manual custom combinations unless there is a strong reason.

A cipher suite is a named combination of algorithms used together to establish a secure connection, most visibly in TLS (Transport Layer Security). A TLS cipher suite specifies: the key exchange algorithm, the authentication algorithm, the bulk encryption cipher and mode, and the MAC algorithm.
Example: TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384 means:
- Key exchange: ECDHE (Elliptic Curve Diffie-Hellman Ephemeral)
- Authentication: RSA
- Cipher: AES-256-GCM
- MAC/Hash: SHA-384

"Ephemeral" key exchange (DHE, ECDHE) is important because it provides forward secrecy: even if the server's long-term private key is later compromised, past session keys cannot be derived from it.\ 
Sessions with non-ephemeral key exchange (e.g., plain RSA key exchange) do not have this property — a recorded past session could be decrypted in the future if the private key is stolen.\
What to configure:

- Use TLS 1.2 or TLS 1.3. Disable TLS 1.0 and TLS 1.1; they are deprecated.
- Disable SSL 2.0 and SSL 3.0 entirely.
- Prefer ECDHE cipher suites for forward secrecy.
- Prefer GCM cipher modes (authenticated encryption).
- Disable RC4, DES, 3DES, and export-grade cipher suites.

### Algorithm strength

Not all algorithms provide the same level of security.

Strength depends on factors such as:

- Algorithm design
- Key length
- Mode of operation
- Implementation quality
- Resistance to current attacks
- Current industry guidance

Common pitfalls include:

- Using outdated algorithms such as MD5 or SHA-1 for security-sensitive purposes
- Using DES or weak 3DES configurations
- Using RSA keys that are too small
- Using ECB mode for block ciphers
- Assuming an algorithm is secure forever
- Reusing keys or nonces in unsafe ways

Developers should use approved modern algorithms and avoid inventing custom combinations.

Key practical pitfalls:

#### Inventing your own cryptography. 
Do not design custom encryption algorithms or modify standard ones. Cryptographic design is an academic discipline requiring years of peer review. Even subtle deviations from standard implementations introduce vulnerabilities. Use well-audited libraries.
#### Using ECB mode. 
AES in Electronic Codebook (ECB) mode encrypts each block independently. Identical plaintext blocks produce identical ciphertext blocks, leaking structural information about the data. Never use ECB for anything. Use GCM or another AEAD mode.
#### Reusing IVs or nonces. 
Initialization vectors (IVs) and nonces must be unique for each encryption operation. Reusing a nonce with a given key in GCM mode is catastrophic and can expose the key. Always generate IVs randomly using a cryptographically secure random number generator.
#### Using a non-cryptographic random number generator. 
Standard library random functions (e.g., Math.random() in JavaScript, random.random() in Python) are not cryptographically secure. For key generation, nonce generation, token generation, and anything security-sensitive, use the platform's cryptographically secure random number generator (crypto.randomBytes(), os.urandom(), SecureRandom, etc.).
#### Encoding as encryption. 
Base64 encoding is not encryption. It is a reversible encoding scheme with no secret key. Data that is Base64-encoded is as accessible as plaintext.

### Misuse of encryption modes

Even when the cipher itself is strong, insecure mode selection can create vulnerabilities.

Examples:

- ECB reveals patterns in plaintext
- Reused nonces can break certain schemes
- Using encryption without integrity protection can allow tampering

Modern authenticated encryption modes are preferred where available because they provide confidentiality and integrity together.

Examples include:

- AES-GCM
- ChaCha20-Poly1305

### Confusing encoding with encryption

Encoding transforms data for representation or transport. It is not secrecy.

Examples of encoding:

- Base64
- Hexadecimal
- URL encoding

A common mistake is treating encoded data as protected data.

### Rolling your own cryptography

Developers should not design new cryptographic algorithms or custom security constructions unless they are qualified cryptography experts working in a formal research or standards context.

Common risks include:

- Weak key derivation
- Broken randomness assumptions
- Missing integrity checks
- Unsafe protocol flows
- Hidden side-channel weaknesses

The safe rule is:

**Use well-vetted algorithms, libraries, and protocol designs.**

## 5.4 Attacks Against Cryptographic Solutions

Cryptographic systems may fail because of weak algorithms, weak keys, poor implementation, bad configuration, poor key handling, or incorrect trust assumptions.

### Brute-force attacks

A brute-force attack tries many possible keys, passwords, or secrets until the correct one is found.

The feasibility depends on:

- Secret length
- Secret entropy
- Rate limits
- Hashing cost
- Available computing power

Examples:

- Guessing weak passwords
- Trying possible encryption keys
- Guessing short API secrets

Defenses include:

- Strong keys
- Long, random secrets
- Password hashing
- Rate limiting
- MFA where appropriate

A brute force attack exhausts all possible keys or passwords until the correct one is found. Its feasibility depends entirely on the size of the search space.\
Against encryption keys, brute force is effectively infeasible for modern key lengths: a 128-bit AES key has 2^128 possible values. No realistic amount of computing power can search this space.\
Against passwords and low-entropy secrets, brute force is a serious threat. If an attacker obtains a hashed password database, they can hash candidate passwords and compare them to the stored values offline, at rates of billions of attempts per second on modern hardware. This is precisely why password-hashing functions must be slow (bcrypt, Argon2), and why enforcing minimum password complexity matters.

### Rainbow attacks

Rainbow table attacks use precomputed tables to reverse weak password hashes efficiently.

These attacks are particularly effective when:

- Password hashes are unsalted
- Users choose weak passwords
- Fast hash functions are used for password storage

Defenses include:

- Unique salts
- Slow password hashing algorithms
- Strong password policy
- Breached-password screening

A rainbow table is a precomputed lookup table mapping hash values back to their original inputs, trading storage space for computation time. Given a list of stolen password hashes, an attacker can simply look up each hash in a rainbow table rather than cracking each one individually.\
The complete defense is salting. A salt is a unique, random value prepended or appended to the password before hashing. Because the salt is different for every user, precomputed tables are useless — an attacker would need a separate table for every possible salt value, which is computationally infeasible.\
Properly implemented password hashing functions (bcrypt, Argon2) incorporate salting automatically. If you are using a general-purpose hash like SHA-256 for passwords, you are doing it wrong regardless of salting.

### Collusion attacks

Collusion attacks occur when multiple parties cooperate in a way that breaks expected trust or access controls.

The exact meaning depends on the system, but examples include:

- Multiple insiders combining their access
- Participants sharing cryptographic material improperly
- Threshold or group systems being undermined by cooperating members
- Key escrow or delegated trust being abused

Defenses depend on the design, but may include:

- Separation of duties
- Threshold controls
- Audit trails
- Hardware-backed protection
- Careful trust modeling

A collision attack finds two different inputs that produce the same hash output. This breaks the collision resistance property.\
In practice, collisions can be exploited to forge digital signatures, create malicious software updates that match a legitimate hash, or undermine certificate trust. SHA-1 is fully broken against collision attacks — demonstrated real-world collisions have been produced (SHAttered, 2017). MD5 was broken earlier still.\
The mitigation is to use algorithms with no known collision attacks: SHA-256 and above.\
Length Extension Attacks\
A length extension attack exploits a structural weakness in certain hash constructions (specifically those using the Merkle–Damgård construction, including SHA-1 and SHA-256). If you know H(message) and the length of message, you can compute H(message || extension) without knowing message. This breaks naive MAC constructions of the form H(secret || message).\
The mitigation: Use HMAC rather than constructing MACs manually. HMAC is specifically designed to resist length extension attacks. Alternatively, SHA-3 is not vulnerable to this attack by design.

### Replay attacks

A replay attack happens when a valid cryptographic message is captured and resent later to repeat an action or impersonate a legitimate request.

Examples:

- Reusing an authentication token
- Replaying a signed request
- Resubmitting a captured protocol exchange

Defenses include:

- Nonces
- Timestamps
- Sequence numbers
- Short-lived tokens
- Replay detection mechanisms

### Man-in-the-middle attacks

A man-in-the-middle attacker intercepts communications between parties and may observe, alter, or inject data.

Cryptography helps defend against this only if identity verification is correct.

Defenses include:

- Correct certificate validation
- Strong TLS configuration
- Trusted key exchange
- Certificate pinning in carefully chosen cases

### Downgrade attacks

A downgrade attack forces a system to use weaker cryptographic options than intended.

Examples:

- Falling back to old TLS versions
- Negotiating weaker algorithms for compatibility
- Accepting unsigned or weakly signed artifacts

Defenses include:

- Disabling obsolete protocols
- Enforcing secure minimum versions
- Rejecting insecure fallback behavior

In a downgrade attack, an attacker interferes with a protocol negotiation to force both parties to use a weaker, deprecated algorithm than they would otherwise choose — one the attacker can then break.\
The mitigation: Disable deprecated protocol versions and cipher suites on the server. If the server refuses to negotiate weak options, they cannot be forced. HSTS (HTTP Strict Transport Security) and proper TLS configuration prevent protocol downgrade to plain HTTP.

### Side-channel attacks

Side-channel attacks exploit information leaked by implementation behavior rather than directly breaking the algorithm.

Examples of leaked signals:

- Timing differences
- Power consumption
- Memory access patterns
- Error message differences

These attacks matter especially in cryptographic libraries, hardware devices, and high-value systems.

Side-channel attacks extract secret information not by breaking the cryptographic algorithm itself, but by observing its physical or computational behavior: timing (how long an operation takes), power consumption, electromagnetic emissions, or cache behavior.\
Timing attacks are the most relevant to developers. A comparison function that returns early when it finds a mismatch leaks information about the secret — an attacker can measure response times to determine how many characters of a guess were correct.\
The mitigation: Always use constant-time comparison functions provided by your cryptographic library when comparing secret values (MACs, tokens, hashes). Never use standard string equality functions (==, strcmp) for this purpose.

### Key theft and misuse

In many real incidents, cryptography fails not because the algorithm is weak, but because attackers steal or misuse the key.

Examples:

- Private key leaked from source control
- Secrets exposed in logs
- Unprotected key files on disk
- Compromised build pipeline signing key
- Weak access control on a key vault

This is why key management is as important as algorithm choice.

### Padding Oracle Attacks

In certain cipher modes (notably CBC with PKCS#7 padding), if an application returns different error responses for "decryption failed" versus "decryption succeeded but content is invalid," an attacker can iteratively determine the plaintext by crafting ciphertext and observing error responses.\
The mitigation: Use authenticated encryption (AES-GCM). Authenticated encryption verifies integrity before decryption; tampered ciphertext simply fails. Additionally, always return the same generic error regardless of whether the failure was a padding error or a content error.

## 5.5 Insecure Use of Cryptography

Many systems use cryptography but still remain insecure because it is applied incorrectly.

### Common insecure practices

#### Hardcoding secrets and keys

Embedding keys, passwords, or tokens in source code is a serious risk.

Problems include:

- Secrets leaking through repositories
- Exposure in builds or logs
- Difficult rotation
- Reuse across environments

Use secure secret management systems instead.

Hardcoding keys and secrets. Embedding cryptographic keys, passwords, or API secrets directly in source code is a pervasive vulnerability. Source code is often committed to version control, shared with many people, and may be publicly exposed. Keys hardcoded in mobile applications can be extracted by decompiling the binary. Keys belong in dedicated secret management systems.

#### Using outdated or weak algorithms

Examples include:

- MD5 for integrity-sensitive use
- SHA-1 in new security designs
- DES or obsolete ciphers
- Weak RSA key sizes

Developers should follow current guidance and framework defaults.

#### Reusing nonces, IVs, or keys unsafely

Some cryptographic schemes become insecure if values expected to be unique are reused.

This is especially dangerous in authenticated encryption and stream cipher contexts.

#### Encrypting without integrity protection

Encryption alone may hide content but still allow tampering.

Use authenticated encryption or pair encryption with integrity protection.

#### Storing passwords with fast hashes

General-purpose hash functions are too fast for password protection.

Do not store passwords using plain:

- SHA-256
- SHA-1
- MD5

Use password hashing algorithms designed for credential storage.

#### Disabling certificate validation

Turning off certificate verification during development or troubleshooting is dangerous if the behavior reaches production.

This can completely break TLS trust.

#### Assuming TLS solves everything

TLS protects data in transit, but it does not:

- Fix broken authorization
- Protect data already exposed on the endpoint
- Prevent insecure storage
- Stop misuse of plaintext after decryption
- Protect against compromised clients or servers

#### Using cryptography without understanding trust boundaries

Examples:

- Signing tokens but failing to validate issuer or audience
- Encrypting sensitive data while exposing keys to all services
- Assuming internal networks are trusted and skipping verification
- Sharing one key across too many systems

Cryptography should be designed around clear trust boundaries and least privilege.

#### Inventing custom security schemes

Homemade encryption formats, token designs, or challenge-response flows are often fragile and unsafe.

Developers should prefer standard, reviewed constructions and protocols.

### Rolling your own cryptography. 

This encompasses writing custom ciphers, modifying standard algorithms, combining primitives in novel ways, or implementing standard algorithms from scratch rather than using a vetted library. The probability that a developer produces a secure custom implementation is extremely low. Use high-level cryptographic libraries (libsodium, the platform's built-in crypto APIs) that make it difficult to make low-level mistakes.

### Using deprecated or weak algorithms. 

MD5, SHA-1, DES, RC4, and TLS 1.0/1.1 appear regularly in code written before their weaknesses were widely understood. Auditing your dependency tree and TLS configuration for these is a necessary ongoing activity.

### Insecure random number generation. 

Using Math.random(), rand(), or similar for anything that needs to be unguessable (session tokens, CSRF tokens, password reset links, encryption keys) is a critical vulnerability. These generators are designed for statistical properties, not unpredictability.

### Incorrect IV/nonce handling. 

IVs must be random and unique per encryption. Generating them from predictable values, reusing them, or embedding them in the ciphertext in a way that allows an attacker to predict the next one all lead to vulnerabilities.

### Skipping certificate validation. 

Disabling TLS certificate verification — a common shortcut to "fix" a TLS error during development — eliminates the authentication guarantee of TLS entirely. The connection may be encrypted, but you cannot know who you are talking to. This code pattern must never reach production.

### Storing passwords with reversible encryption. 

Passwords should never be stored in a form that can be recovered as plaintext, even by the system itself. If a data breach occurs, the attacker should not be able to recover users' passwords. Use one-way password hashing (Argon2id, bcrypt) — the application only needs to verify a supplied password against the stored hash, it never needs to recover the original.

## 5.6 Key Management

Key management is the set of processes used to handle cryptographic keys securely throughout their lifecycle.

Strong algorithms do not provide meaningful security if keys are generated poorly, stored insecurely, shared too broadly, or never rotated.

### Key generation

Key generation must use a cryptographically secure random source.

Good key generation practices include:

- Use approved libraries or managed services
- Generate keys with sufficient length and strength
- Separate key purposes
- Avoid predictable or user-derived key values unless a secure derivation function is used
- Record ownership, purpose, and lifecycle metadata

Different keys may be needed for:

- Encryption
- Signing
- Transport protection
- Key wrapping
- Authentication
- Backup recovery

Keys must be generated with a cryptographically secure random number generator (CSPRNG). The operating system provides CSPRNGs (/dev/urandom on Linux/macOS, BCryptGenRandom on Windows, crypto.randomBytes() in Node.js, secrets module in Python). Never use general-purpose random number generators.\
Key length must meet current security recommendations (see section 5.3). Longer keys are not always better — some algorithms have optimal key sizes — but below-minimum key sizes are always insecure.\
For asymmetric keys, use the key generation utilities provided by your cryptographic library. Do not attempt to construct key pairs manually.

### Key storage

Keys should be stored in secure locations with strict access control.

Preferred approaches include:

- Hardware security modules (HSMs)
- Cloud key management services
- Managed secrets systems
- OS-protected keystores
- Encrypted key containers with strong access control

Avoid:

- Source code repositories
- Plaintext configuration files
- Shared folders
- Logs
- Unprotected backups

Private keys and long-lived secrets should receive stronger protection than public data or temporary values.

The most common key management failure is storing keys alongside the data they protect, or in source code, configuration files, or databases without additional protection.\
Options for key storage, in rough order of security:
- Hardware Security Modules (HSMs): Dedicated hardware devices that store keys in tamper-resistant memory and perform cryptographic operations internally. The key never leaves the hardware. Used for the highest-security requirements.
- Cloud Key Management Services (KMS): AWS KMS, Google Cloud KMS, Azure Key Vault. Keys are managed by the cloud provider and operations are performed via API. Keys are hardware-backed and never exposed directly to the application.
- Secrets management systems: HashiCorp Vault, AWS Secrets Manager. Keys and secrets are stored centrally, access is controlled and audited, and secrets are delivered to applications at runtime.
Environment variables and secure configuration: For lower-stakes keys, environment variables keep secrets out of source code. This must be combined with OS-level access controls and must not log environment variables.

Never store keys in: source code, version control repositories, application databases, log files, or unprotected configuration files.

### Key exchange

Key exchange is the secure establishment or transfer of keys between parties.

This must be done in a way that prevents interception or substitution.

Examples:

- TLS handshake key establishment
- Public key exchange with certificate validation
- Secure secret distribution from a central vault
- Provisioning device credentials during enrollment

Developers should avoid ad hoc methods such as emailing keys or copying secrets manually unless wrapped in an approved secure process.

Key exchange is the process of establishing a shared secret between two parties over an insecure channel. The Diffie-Hellman (DH) and Elliptic Curve Diffie-Hellman (ECDH) protocols allow two parties to derive an identical shared secret without ever transmitting it. An observer sees the exchange but cannot derive the shared secret.\
In TLS, ephemeral variants (DHE, ECDHE) are used so that a new key pair is generated for each session. This provides forward secrecy — compromise of the server's long-term private key does not expose past sessions.\
Asymmetric encryption (RSA) can also be used for key exchange: the sender encrypts a randomly generated symmetric key with the recipient's public key, and only the recipient can decrypt it. This is simpler but does not provide forward secrecy.

### Key rotation

Key rotation replaces an existing key with a new one.

Rotation may be required because:

- A key is aging
- A policy requires regular renewal
- A compromise is suspected
- An employee leaves
- A certificate is nearing expiry
- Cryptographic guidance changes

Good rotation practices include:

- Support overlap between old and new keys where needed
- Automate rotation when practical
- Track which systems use which keys
- Re-encrypt or re-sign data safely as needed
- Log key lifecycle events

Key rotation is the practice of periodically replacing cryptographic keys with new ones and re-encrypting data under the new key. Rotation limits the exposure window if a key is silently compromised and reduces the amount of data exposed if a key is eventually broken.
Rotation practices:
- Define a maximum key lifetime and enforce it. Different key types and risk levels warrant different rotation intervals.
- Automate rotation wherever possible. Manual rotation is error-prone and is often deferred.
- Maintain a key versioning scheme. Data must remain decryptable during the transition period between old and new keys. Include the key version identifier with encrypted data so the correct decryption key can be selected.
- After rotation, re-encrypt all data protected by the old key under the new key. The old key should eventually be retired.

### Key deletion

Key deletion means securely removing a key that is no longer needed.

Deletion should be controlled and documented because it may make protected data unrecoverable.

Developers and operators should understand:

- Which data depends on the key
- Whether backup recovery is required
- Whether deletion is immediate or scheduled
- Whether the environment supports secure destruction guarantees

When a key is no longer needed, it must be securely and irrecoverably destroyed. This is not simply deleting a file.
Secure deletion considerations:

- Keys in memory must be explicitly zeroed out after use. Languages with garbage collection may move objects in memory; use secure memory APIs where available.
- Keys stored on disk may persist in filesystem metadata, swap space, log files, backups, or version control history even after the file is deleted. Deletion from a KMS or HSM guarantees secure destruction; filesystem deletion typically does not.
- Encryption keys that are irrecoverably deleted render the data they protected permanently inaccessible. Verify that all necessary re-encryption or data migration has occurred before destroying a key.
- Document key decommissioning events as part of your audit trail.

### Key decommissioning

Key decommissioning is the structured retirement of a key from active use.

This may include:

- Marking it inactive
- Removing it from applications
- Revoking dependent credentials
- Replacing trust relationships
- Confirming no systems still depend on it
- Archiving metadata for audit purposes

Decommissioning is broader than deletion. A key may first be retired operationally, then deleted later under a governed lifecycle.

Key decommissioning is the broader process of retiring a key from active use: marking it as deprecated, removing it from active key distribution, and ensuring systems no longer accept data encrypted under it, before final destruction. A key should be decommissioned before it is deleted to allow for an orderly transition.

### Key management principles

Developers should remember these core practices:

- Use separate keys for separate purposes
- Minimize who and what can access keys
- Prefer managed key services where possible
- Monitor key access and use
- Rotate keys when needed
- Revoke compromised keys quickly
- Protect backups that contain keys
- Document ownership and lifecycle
- Avoid manual secret handling when automation exists

Key Access Control and Auditing
Key management also encompasses controlling who and what can use a key.

- Apply the principle of least privilege: each service or component should have access only to the keys it requires for its specific function.
- Log all key access and operations. If a key is used unexpectedly, the audit trail is the mechanism for detecting it.
- Separate key management duties: the person or system that generates keys should not be the same as the one that accesses the data they protect where feasible.
- Use distinct keys for distinct purposes. Do not reuse a key across different systems, contexts, or data types. If one system is compromised, key reuse spreads the blast radius.

## 5.7 Practical Developer Guidance

Developers often interact with cryptography through frameworks, cloud services, and libraries rather than raw algorithms. Even so, they must understand the security impact of the choices they make.

Practical rules:

- Prefer standard libraries over custom implementations
- Use modern, recommended algorithms and modes
- Rely on secure defaults where possible
- Protect secrets and private keys from exposure
- Use parameterized APIs and high-level protocol libraries
- Validate certificates and signatures correctly
- Use password hashing for passwords, not general hashing
- Use authenticated encryption for sensitive data where appropriate
- Review cryptographic settings during design and deployment
- Treat key management as a core security responsibility

Cryptography is a tool, not a solution. Its effectiveness depends on selecting the right primitive for the right purpose, applying it correctly, and managing the resulting keys with operational discipline.

- Match the tool to the security property. Confidentiality needs encryption; integrity needs MACs or AEAD; authentication needs signatures or certificates; password storage needs password-hashing functions.
- Use modern, vetted algorithms. AES-256-GCM, SHA-256+, RSA-2048+, Argon2id, TLS 1.2/1.3. Avoid MD5, SHA-1, DES, RC4, ECB mode, and TLS 1.0/1.1.
- Never invent your own cryptography. Use high-level libraries. The number of ways to misuse low-level primitives far exceeds the number of ways to use them correctly.
- Use CSPRNGs for everything unpredictable. Keys, nonces, IVs, tokens, salts — all require a cryptographically secure source of randomness.
- Authenticate your ciphertext. Encryption without integrity protection is vulnerable to tampering. Use AEAD modes or Encrypt-then-MAC.
- Manage keys as a first-class security concern. Store them in dedicated systems, rotate them on schedule, control and audit access, and destroy them securely when decommissioned.

## 5.8 Testing and Reviewing Cryptographic Controls

Cryptographic controls should be tested and reviewed as security-critical functionality.

### What to review

- Which algorithms are used
- Which key lengths are used
- How keys are generated
- Where keys are stored
- Who can access the keys
- Whether rotation and revocation are supported
- Whether certificate validation is enforced
- Whether password storage uses proper password hashing
- Whether encryption also provides integrity where needed
- Whether secrets appear in logs, code, or configuration

### What to test

- Correct encryption and decryption behavior
- Correct signature generation and verification
- Rejection of expired or invalid certificates
- Rejection of tampered tokens or data
- Password verification correctness
- Rotation and decommissioning processes
- Access control around keys and secrets
- Failure handling when keys are unavailable or revoked

### Operational checks

Also verify:

- Key access is logged
- Secret scanning is in place
- Certificates are renewed before expiry
- Compromised keys can be revoked quickly
- Recovery processes exist for critical keys
- Cryptographic dependencies are kept current

## Key Exam Takeaways

Developers should remember these core points:

- Cryptography helps provide confidentiality, integrity, authenticity, and sometimes non-repudiation.
- Keys are central to cryptographic security and must be managed carefully.
- Hashes and message digests help protect integrity but do not by themselves prove origin.
- X.509 certificates bind public keys to identities and support trust in TLS and PKI systems.
- Digital signatures help verify origin and integrity.
- Outdated algorithms, weak cipher choices, and bad modes can break security.
- Rainbow attacks and brute-force attacks are especially relevant to weak password storage and low-entropy secrets.
- Many cryptographic failures are really key management failures.
- Common insecure practices include hardcoded keys, weak hashing, unsafe certificate handling, and custom cryptographic design.
- Key generation, storage, exchange, rotation, deletion, and decommissioning must all be handled securely.
- Developers should use well-vetted libraries, secure defaults, and managed services whenever possible.
